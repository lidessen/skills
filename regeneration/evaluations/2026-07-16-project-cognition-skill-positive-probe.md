## Claim and fixed comparison

The approved [form decision](../../design/organization/sessions/2026-07-16-project-cognition-form-decision.md)
selected an independently installable domain Skill plus a conditional,
rebuildable target-project projection. The positive claim was:

> A fresh agent using only the installed Skill and a named project can choose a
> project-specific decomposition, create a source-linked reusable carrier, and
> improve a later decision without inheriting the conversations that formed the
> method.

The target expression selected P05 as Primary and P12, P13, and P14 as supports
in the [expression-team record](../../design/organization/sessions/2026-07-16-project-cognition-expression-team.md).
The first [Skill expression](../../skills/project-cognition/SKILL.md) retained
bootstrap, refresh, and verify paths, adaptive decomposition, a small Markdown
projection contract, and a standalone generated Sequence snapshot. It added no
runtime, database, fixed facet taxonomy, hard file-size limit, or Shilu
dependency.

`semajsx-v0` supplied a structurally different comparison from the earlier Go
and SQLite [Shilu probe](2026-07-16-project-cognition-bootstrap-and-refresh.md):
169 tracked files in a TypeScript/TSX package monorepo with several rendering
targets, a style contract, hydration, signal state, public re-exports, and
package-local verification.

The original dirty `semajsx-v0` checkout was excluded. A clean detached
worktree at `8240e79` was copied into isolated baseline and treatment
workspaces. Both received the same model, source, raw task, acceptance, tools,
write scope, estimated work, and completion contract. The named future
decisions were adding a rendering target, evolving style or hydration
contracts, and tracing cross-package impact. Only the treatment workspace
contained the packaged Skill.

Evaluator run `8228f68b-6216-4316-afb1-98a62b4ef3c5` passed and submitted

- both carriers recovered the principal renderer and style relations;

This supports the Skill expression on one structurally different project. It
does not prove a universal ontology, complete project coverage, or that every
project should retain a cognition directory.

## Authority correction

Human inspection found a boundary error that the blind source judge did not
report. The treatment index said:

```text
Verifier: project-cognition Skill (bootstrap path)
```

That makes the generating method appear to verify its own projection. It
violates P13/P14 even though the same index also called itself
non-authoritative. The Skill was narrowed so that a verifier must be an
accountable external actor, review run, or governed process; a bootstrap
self-check remains `proposed` or `unverified`.

Focused run `c047bb5c-3e5f-425d-8dc2-fbb99eadc039` loaded the revised Skill and
old candidate, cited the exact verifier wording, returned
`correction_required`, and proposed the minimum truthful form:

```text
Status: non-authoritative, rebuildable, unverified
Verification: bootstrap self-check only
Future verifier: named separately from retention/commit owner
```

The correction changed only the Skill action and projection instructions. No
acceptance runtime or new record type was added. This probe also shows that a
source-accuracy judge alone is insufficient for an authority-expression claim;
the boundary needs its own check.

## Decision

Deploy the corrected Skill as supported for bounded use.

The supported boundary is:

```text
project-cognition Skill -> decide temporary vs durable cognition,
                           adapt semantic decomposition,
                           reconstruct and refresh a projection
target project source   -> fact authority
external verifier       -> source-check named decision field
Work Cell               -> optional bounded execution and raw evidence
orchestration           -> optional release and settlement
context-engineering     -> optional delivery of selected source/projection
```

Reopen or withdraw the Skill if another project cannot choose meaningful cuts
without hidden author knowledge, a one-off task repeatedly creates durable
files, the host cannot reliably activate and deliver the Skill, projections
commonly name their generator as verifier despite the correction, most small
changes invalidate the whole carrier, or maintaining cognition costs more than
direct reconstruction for its declared decisions.
