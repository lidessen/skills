# Context Engineering Rewrite Probe

## Claim

Replacing `harness` with `context-engineering` should preserve the useful
runtime-evidenced delivery judgment while narrowing its unit of work from a
project-wide harness architecture to one authoritative source-to-agent action
path. It should route content authoring, artifact authority, skill expression,
and runtime implementation to their existing owners.

Three fresh read-only agents received raw tasks. The action and boundary probes
exercise the new form. The inherited/new comparison is an instruction-level
trace, not a matched behavioral experiment; no causal improvement claim is made.

## Action probe

**Task:** audit how the requirement to read the Sequence and selected P-ID
interpretations before materially updating a skill reaches a Codex agent in this
repository. Do not edit.

**Disconfirming observation:** the agent assumes an instruction filename loads,
builds a generic L1/L2/L3 architecture, creates a durable evaluation by default,
or cannot distinguish the semantic source from its delivery projection.

**Observed action:** the agent identified
[`principles/SEQUENCE.md`](../../principles/SEQUENCE.md) as semantic authority,
the selected interpretations as governed derivatives, and root
[`AGENTS.md`](../../AGENTS.md) as the Codex-facing delivery projection. It
observed that this task's prompt contained the root instructions before scoped
action, then followed the activated skill to the host Sequence and selected
interpretations. It kept the internal loader trace and behavior of another
Codex surface or version explicitly unknown.

The agent selected no placement change. It proposed an ordinary material skill
update in a disposable context as the smallest stronger probe and declined to
create a durable artifact for the bounded audit.

**Verdict:** supported for action, source/projection distinction, restraint, and
task-specific path observation. General runtime attribution remains open.

## Boundary probe

**Task:** add an important new retrieval-ranking decision rule to the P09
interpretation. Do not edit.

**Disconfirming observation:** the skill writes or accepts doctrine because the
future rule would eventually need delivery.

**Observed action:** the agent classified the request as missing or unaccepted
content, not failed delivery. It routed the inquiry to
[`principle-cultivation`](../../skills/principle-cultivation/SKILL.md) and
distinguished a project-local retrieval practice, a clarification already borne
by P09, and an irreducible Sequence candidate. It left context engineering until
after any content becomes accepted.

**Verdict:** supported.

## Inherited/new comparison

**Scenario:** a coding agent repeatedly ignores an architecture boundary already
documented in the project; determine the smallest response.

Both forms begin by checking actual delivery rather than rewriting doctrine or
installing a runtime guard. The inherited `harness` then expands to a runtime
capability map, project L1/L2/L3 and handoff inventory, durable evaluation home,
paired comparison, boundary probe, and path observation. That remains useful
for the former project-wide object, but it is broader than the scenario.

The new form begins with the failed judgment and source, traces only the
delivery relation that could change it, and permits a conversation result. If
the source already arrives, it stops treating the problem as delivery and routes
the expression or action failure. It makes delivery projections subordinate to
the source and maps timing to runtime-native surfaces rather than mandatory
containers.

**Verdict:** the new expression is the smaller truthful form for the tested
action-level scenario. The comparison is self-evaluated instruction evidence,
not independent proof of improved agent behavior.

## Migration and deployment decision

Keep `context-engineering` active with P09/P02/P14/P08. Retire the active
`harness` path and `init` command without a compatibility alias. Preserve old
decisions and evaluations where `harness` names the historical object.

The next behavior probe should use an ordinary material skill update without an
explicit context-engineering reminder and observe whether the selected Sequence
path is followed before the first mutation. Reopen the boundary if real tasks
repeatedly require context engineering to own doctrine, artifact lifetime, or
runtime implementation in order to produce a useful result.
