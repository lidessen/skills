# Systems Engineering First-Slice Probe

**Date:** 2026-07-19

**Status:** first-slice boundary supported; consequential direct output remains guarded

**Execution profile:** OpenCode Go `deepseek-v4-flash`, AI SDK v7 Work Cell,
read-only repository access, required terminal settlement

**Authority:** evaluation evidence only; no run accepted a project decision

## Question

Can the proposed [`systems-engineering` Skill](../../skills/systems-engineering/SKILL.md)
help an economical Flash-class Agent distinguish proportional engineering from
process inflation, identify a whole-system control gap before optimizing local
Cells, and reserve higher assurance for a high-consequence case?

The probe also tested the decision derived from [research on engineering
reliability with fallible components](../../principles/research/engineering-reliability-from-fallible-components.md):
the desired object is sufficiently reliable whole-system behavior under
disturbance, not semantic infallibility of one Cell.

## Contrasted cases

Three independent read-only Cells received the same installed Skill:

1. a reversible README link error already observed by a deterministic CI link
   checker;
2. the retained five-Cell review case in which packet-local absence became a
   false whole-project claim, host verification later filtered the noise and
   confirmed one real O(n) defect, and usage reached 2.57 million tokens or
   5.47 times the estimate; and
3. a high-consequence multi-tenant billing migration with documented source
   invariants, available deterministic checks and dry run, possible backups and
   staged rollout, and a human billing owner holding production approval.

Each Cell had to settle one structured System Case. The first manifest forecast
85,000 tokens across the three Cells.

## Direct-Cell observations

All three Cells settled on each of three iterations. The simple case was stable
across all runs: it retained the existing CI check and Git recovery path and
explicitly rejected an additional Swarm, review stage, or permanent artifact.
This supports the proportionality boundary; the Skill did not turn systems
engineering into a mandatory workflow.

The review case consistently found the correct principal relation: protocol
settlement and packet-local findings were being promoted to whole-project facts
without resolution against the whole source. It also kept the 5.47-times usage
variance as a post-run estimation signal rather than reinstating a hard runtime
token stop. It did not recommend more reviewers.

The migration case consistently reserved production authority for the human
billing owner and selected independent observation, staged effects, and
recoverable rollback as relevant control properties. It did not claim that the
Agent or reviewers could become perfect.

The direct candidates were not acceptable final System Cases. Flash repeatedly
invented plausible precision even after the prompt gained an explicit
authority-and-evidence gate. Examples included:

- an `evidenceScope` terminal field and an in-run escalation threshold in an
  early review candidate;
- later review thresholds of greater than two or three times estimate and a
  three-run reopening window;
- billing sample sizes, assertion types, monitoring periods, a two-times
  dispute-rate threshold, and organization roles not supplied by the case; and
- boundary statements claiming that no such details had been introduced.

This is decision-changing evidence. A complex prompt cannot make a
consequential open-ended systems design a reliable primitive for this reference
profile. The Skill therefore treats its first result as a candidate and uses an
independent narrow claim/source check for newly introduced specifics.

## Independent specificity verification

Two verifier Cells received one fixed judgment only: classify each introduced
mechanism, field, threshold, duration, sample, protocol, and role as
source-supported, owner-to-determine, or unsupported invention. They were not
allowed to redesign the system.

The first verifier run read the complete Cell run records. Both verifiers
correctly returned `fail` and identified the material unsupported values and
roles. However, they also inspected rejected intermediate tool calls rather
than only the accepted terminal payload. The run used 109,694 tokens against a
50,000-token forecast, a 119.4% relative error. Full execution traces were the
wrong verification context.

The caller then projected only the original case and accepted terminal payload.
The unchanged two-verifier shape again returned `fail` and preserved the
supported requirements while rejecting the unsupported thresholds, durations,
mechanisms, and roles. It used 58,054 tokens against the same 50,000-token
forecast, a 16.1% relative error. Compared with full records, total usage fell
47.1% and input usage fell from 103,097 to 50,844 tokens. The compact projection
also removed the rejected-call contamination.

The verifier is still fallible. In the migration case it was too willing to
accept some plausible mechanisms as supported by a more general source
capability. Its output therefore remains a claim for host or source settlement,
not authority. The result supports bounded independent verification, not a
promise of perfect quality control.

## Corrections retained

The Skill now:

- distinguishes a required control property from its concrete representation;
- forbids unsourced fields, thresholds, durations, samples, protocols, and
  roles while allowing `[owning domain/runtime to determine]`;
- keeps resource-estimate variance as post-run learning by default;
- treats self-audit as non-verification for a consequential System Case; and
- directs the independent verifier to a compact accepted candidate plus its
  governing sources, not a full raw execution trace.

No Work Cell core field, terminal protocol, domain review packet, or runtime
budget mechanism was added. The finding is presently a harness strategy in the
Skill and evaluation layer.

## Verdict

**Supported:** the first slice expresses a distinct whole-system judgment,
declines unnecessary controls on a simple case, finds the correct integration
gap in a multi-Cell review system, and allocates assurance proportionally in a
high-consequence case. Compact independent verification materially improves
the candidate-to-source control path and context economy.

**Not supported:** a single Flash Cell cannot yet produce a consequential final
System Case with stable specificity. The candidate plus narrow verifier plus
host/source settlement is the supported form. Operation evidence is still
required before any concrete system is called reliable.
