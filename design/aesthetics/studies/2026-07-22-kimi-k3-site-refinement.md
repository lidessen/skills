# Aesthetic Study — Kimi K3 Design Distillation

**Status:** controlled distillation completed; current site treatment accepted for release while the abstract relation remains provisional
**Case:** [Project Site Pilot](../cases/2026-07-14-project-site-pilot.md)
**Method:** [`visual-design`](../../../skills/visual-design/SKILL.md)
**Execution profile:** Kimi K3 through Kimi Code 0.28.1

## Question

The first use of Kimi treated it as an art-direction critic while another agent
implemented the result. That did not answer the actual question: **what does
Kimi do while designing and implementing, and which parts of that observable
process should become a portable method?**

This study does not request or infer hidden chain of thought. It compares
observable repository reads, tool calls, mutations, rendered inspections,
revisions, cleanup, and verification.

Kimi's own product description supports studying the whole execution profile,
not the bare model. [Kimi Websites](https://www.kimi.com/help/websites/websites-overview)
describes a dedicated `webapp-building` skill that moves through analysis,
planning, stack and asset selection, code construction, and repeated preview
optimization. [Kimi Code](https://www.kimi.com/resources/kimi-code-introduction)
likewise describes repository analysis, planning, command execution, and
iteration. The public [Kimi Code repository](https://github.com/MoonshotAI/kimi-code)
exposes the general agent harness, but not the private Websites skill.

## Controlled setup

Four isolated workspaces began at commit
`d9a93c200bc1f8b3176f86ae985c2f729d1e9039`. Each received the same site task,
model route, repository content, and completion condition: implement the real
site, build it, inspect desktop and mobile rendering, revise, and report only
after rendered inspection. The requested relation was a faint public and
communal afterimage with no direct symbol, propaganda anatomy, retro display,
or political slogan; the sensory cue was dried citrus—warm, slightly bitter,
and retaining a small lift.

The changing treatment was the only intended variable:

1. Kimi Code and K3 without the project skill;
2. the then-current `visual-design` skill;
3. a first candidate that moved rendering earlier and made the artifact loop
   explicit; and
4. a second candidate that also distinguished preservation of content and
   accepted relations from preservation of the current visual form, while
   removing the component-expression reference from default design context.

Counts come from exported Kimi Code session ledgers. They are descriptive, not
a quality score: more calls are useful only when they close the artifact loop.

### Native strength

The native run behaved as an autonomous visual implementer. It read source,
formed a task list, repaired its isolated dependency environment, built a
multi-route screenshot probe, inspected renders repeatedly, noticed stale or
blank capture paths, corrected title wrapping and overlapping details, removed
an accidentally added dependency, and ran build and link checks. It also made
the largest concrete design move: a new public-workbench card and richer
component anatomy.

That amplitude was not unconditionally good. The public notice number, hanging
archive stamp, pushpin, and paper grain made the requested historical remainder
more literal. The model's strength needs a small negative boundary; it does not
need a large design textbook in front of it.

### How the skill suppressed the model

The original skill made Kimi read the main prompt, four interpretations, a
command, visual-language guidance, component-expression guidance, and an
exhaustive delegated project report before its first mutation. Its language of
visual contracts, conformance maps, invariants, and consumers improved caution
but shifted the task from making to proving compliance. Kimi interpreted an
accepted direction as an instruction to retain the current composition. The
result withdrew charged colors but did not use its native component invention.

The first candidate proved that an explicit render–observe–revise loop could
restore activity, but not creative authority. It still reported “zero structure
change” as a virtue. The missing distinction was not another aesthetic
principle; it was **what the agent was authorized to preserve and what it was
authorized to redesign**.

### Recovered operating shape

The second candidate said explicitly that content, action, functional
contracts, accepted relations, and negative boundaries remain; current
composition, component anatomy, and treatment do not become invariants merely
because they exist. It also stopped loading the component-expression reference
unless component conformance was the principal unresolved judgment.

Kimi then returned to an implementation-shaped loop: it prepared a treatment,
restored the baseline to render it honestly, reapplied the design, searched all
metadata consumers, rebuilt and restarted a stale preview, inspected real
desktop/mobile routes, revised headline wrapping, verified links, and removed
its temporary screenshot dependency and script.

## Distilled method

The portable contribution is a small control loop rather than a Kimi style:

1. **Thin frame:** give the object, audience action, accepted relation, material
   constraints, and one meaningful negative boundary.
2. **Form authority:** state separately what must remain true and which current
   forms may be replaced. Existing implementation is evidence, not doctrine.
3. **Early artifact:** obtain a representative baseline and make one concrete
   visual wager before exhaustive secondary reading.
4. **Embodied iteration:** implement a real fragment, render it, inspect its
   intended and plausible wrong readings, revise, and only then propagate.
5. **Whole-system sweep:** inspect an ordinary downstream surface and search
   project-wide for consumers of changed roles, assets, and metadata.
6. **Clean proof:** remove probe-only dependencies and run mechanical checks.
   These prove integrity, not beauty.
7. **Human acceptance:** model observations remain proposals; the designated
   human accepts or reopens the perceptual result.

## Project consequence

`visual-design` now keeps its principles and negative-boundary judgment but
does not require a formal conformance artifact before implementation. Its
`design` path starts from a real render, treats redesign as authority to change
visual form, loads component-expression guidance only for a genuinely
component-centered contradiction, requires project-wide consumer coverage, and
returns a compact implementation account.

No experiment arm is automatically promoted into the site. The native arm is
more inventive but too literal; the skill arms are better bounded but still
produce different aesthetic judgments. The main worktree remains the
synthesized project treatment.

## Human acceptance observation

The Principal observed that the revised artifact was more detailed than the
previous version, then identified a flaw in the review question: a faint
historical or communal afterimage is difficult to evaluate directly and is not
something a person can reliably certify by naming its cause. The acceptance
surface was therefore corrected from “did the abstract atmosphere land?” to a
concrete decision: whether the revision is preferable to the current artifact,
whether it creates a clear wrong reading, and whether it is fit for the present
release.

The Principal accepted the current revision under that practical standard. The
decision admits this treatment for release; it does not prove that the original
metaphorical phrase was precisely realized or promote the treatment into a
portable style. The direction remains a project-local hypothesis that later
practice may revise.

## Limits and reopening conditions

This is one model, one harness, one site, and one task repeated once per
treatment. The same model was both executor and observed subject. Reopen the
method if another implementation task still stalls in preflight explanation,
if a redesign again becomes token replacement without rendered justification,
if a freer run returns to costume, or if a different model needs more—not
less—component guidance to act coherently.
