# Aesthetic Case — README Entry

**Status:** revised — human review pending
**Practice source:** [aesthetic practice pilot](../README.md)
**Seed:** [initial preference probe](../seed/2026-07-10-initial-preference-probe.md)

## Concrete object and audience

The object is the repository README's first screen. Its audience is a new
practitioner or potential contributor who needs to decide whether this
repository offers a usable method or bounded evaluation surface for
AI-assisted production work, rather than a generic skill catalog or opaque AI
platform.

An earlier audience description also included an "ordinary user". That was too
wide for this repository's actual role: the project mandate serves ordinary
people, but this repository is currently a methods and evaluation surface, not
an end-user product.

## Direction

| Seed dimension | Translation into this document interface |
|---|---|
| Common Workshop subject | lead with concrete productive capability held in common, not a capability boast |
| Constructive Commons composition | state the repository role first, then route a reader by immediate decision: method, bounded evaluation, or project boundary |
| Living Technology atmosphere | use calm, direct language and leave detail in linked sources rather than overwhelming the first screen |

## Constraints and negative boundary

- Do not claim model parity, universal openness, or a public project name that
  the Principal has not chosen.
- Do not turn the README into a second mandate, design document, or skill
  manual; link to the owning source instead.
- Do not use decorative imagery as proof of the project's social purpose.

## Implementation and acceptance

The README opening must let a reader identify, before the skill catalog:

1. whether the repository serves their practical task of using, adapting, or
   assessing an AI-assisted production method;
2. the current repository's limited role as a methods and evaluation surface;
3. the project's productive-AI purpose without mistaking it for a demonstrated
   product capability; and
4. one next authoritative route for a method, a bounded evaluation, or the
   project boundary.

Human review asks: does this opening make the project more concrete and
reachable without adding hype, confusing the repository with the whole project,
or requiring a reader to reconstruct the navigation?

The bounded AI review recorded in the [README route study](../studies/2026-07-12-readme-route-review.md)
supports this revision but does not replace this human review.

## Reopening observation

Reopen the case if a new practitioner cannot choose a route without reading the
whole page, still mistakes `skills` for the project name, cannot distinguish a
methods repository from an end-user platform, or finds the new opening too
abstract to identify a practical next action.
