# Aesthetic Case — Project Site Pilot

**Status:** approved for matched implementation
**Practice source:** [aesthetic practice pilot](../README.md)
**Form decision:** [project site UI-method pilot](../../decisions/028-project-site-ui-method-pilot.md)
**Seed:** [initial preference probe](../seed/2026-07-10-initial-preference-probe.md)

## Concrete object and audience

The object is the first public home page and documentation shell for this
repository. A new visitor should be able to see a human-scale productive
project, distinguish its purpose from its current `skills` carrier, and reach a
usable method without reconstructing the repository's internal history.

The primary audience is an ordinary technically capable person or small team
looking for practical control over AI-assisted work. Agent-method practitioners
and prospective contributors are secondary audiences. The interface should
respect their intelligence without assuming that they already understand the
Sequence, Work Cell, P-IDs, or the project's organizational metaphors.

## Relation to make perceptible

This is a **common workshop for inspectable productive capability**: concrete
methods, tools, evidence, repair, and contribution are held in view together.
It is neither a venture-backed AI product landing page nor a political poster,
academic philosophy archive, generic documentation theme, or catalog of prompt
tricks.

| Seed dimension | Translation for this case | Avoid |
|---|---|---|
| Common Workshop subject | show work through methods, evidence, tools, and reachable actions; favor a sense of shared making over automated magic | stock AI imagery, glowing brains, robots, dashboards without a user task |
| Constructive Commons composition | use an editorial grid, strong typographic hierarchy, visible structure, and controlled asymmetry to distinguish purpose from routes | a uniform field of rounded cards, decorative metric panels, or symmetry that makes every claim equal |
| Living Technology atmosphere | use warmth, tactility, calm depth, and generous reading rhythm; let blank space carry boundaries | beige nostalgia, ornamental texture behind prose, or nature imagery detached from productive work |

These are relations to test, not a preset palette, font list, component library,
or requirement to use an illustration. Existing accepted seed material may be
proposed when it changes comprehension, but it is not automatically a site
asset.

## Information and interaction hierarchy

The first screen should answer, in order:

1. What common capability is this project trying to build?
2. What is this repository now, and what is it not?
3. What can I inspect or use immediately?

The primary action enters the active methods. A quieter action opens the
founding mandate. Repository and contribution routes remain discoverable
without competing with those first actions. The documentation shell gives
reading position, local navigation, search, source provenance, and an edit or
repository route where truthful.

Do not use an unresolved public project name as the hero. `Skills` may appear
as a plain repository label accompanied by a descriptive line. Do not turn
“liberation,” “commons,” or “open” into a slogan detached from the concrete
replaceability and evidence commitments in the mandate.

## Material constraints

- Start with semantic HTML and useful content before motion or decoration.
- Support narrow mobile, ordinary laptop, and wide desktop layouts without
  hiding essential navigation or introducing horizontal overflow.
- Preserve keyboard reachability, visible focus, reduced-motion preference,
  readable contrast, meaningful headings, and useful alternative text.
- Prefer static delivery and progressive enhancement. No account, analytics,
  remote font, opaque runtime, or client-side framework dependency is required
  to read the first slice.
- Every projected factual page exposes its repository source. Generated content
  and navigation must be reproducible from the checked-in manifest.
- Decorative media must have an attributable source and a named contribution
  to comprehension; otherwise leave the space open.

## Matched implementation brief

Both experimental arms receive this case, the same source manifest, the same
Astro/Starlight starting point, and the same verification contract. Each arm
may establish its own tokens and components, but it may not add content claims,
routes, or functional scope to make its result look more complete. The
treatment may use Impeccable only as declared in the form decision; it may not
silently install persistent hooks or overwrite repository design sources.

The deliverable for each arm is a buildable site, representative screenshots at
the agreed viewports, mechanical check results, resource observations, and a
short account of the design decisions that materially affected the result.

## Human review questions

Review blinded `A` and `B` separately before comparison:

1. Can you identify the project's purpose, this repository's limited role, and
   a useful next action without reading explanatory notes?
2. Does the page feel like a shared, serious place for productive work rather
   than a template, campaign poster, or proprietary AI product?
3. Is the hierarchy confident and memorable while the documentation remains
   calm enough for sustained reading?
4. Which visual or interaction decisions increase trust and comprehension, and
   which are merely polished?
5. Would you ship either direction, combine named parts, or reject both?

Mechanical success admits a result to this review; it never establishes
aesthetic acceptance.

## Study and reopening observation

After implementation, create one study that records the blinded preference,
method reveal, concrete deltas, cost and intervention evidence, rejected
features, and the next smallest change. Do not promote a direction or skill from
the generator's explanation alone.

Reopen this case if visitors still mistake the repository for the whole
project, cannot locate an active method, perceive ideological claim before
practical substance, or if the site needs a name or adoption promise whose
owning source remains unresolved.
