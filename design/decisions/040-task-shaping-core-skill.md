# 040 — Task Shaping as a Core Skill

**Status:** first slice implemented; behavioral claims provisional
**Date:** 2026-07-18
**Authorized by:** principal

## Concrete pressure

Work Cell now provides a generic execution loop, Swarm and orchestration can
release prepared units, and `model-evaluation` can characterize an execution
profile. Domain methods such as code review and project cognition already know
how to form some semantic packets. The missing recurring judgment is upstream
of all of them: whether one concrete task fits the observed stable envelope of
the chosen agent loop, and, when it does not, how to transform the task without
losing the whole that later verification must reconstruct.

The retained [cheap-Swarm cognition probe](../../regeneration/evaluations/2026-07-18-cheap-swarm-cognition-probe.md)
showed that protocol-stable Cells can still produce local semantic errors and
that reconciliation may cost more than formation. The subsequent
[index-first retrieval probe](../../regeneration/evaluations/2026-07-18-index-first-cognition-retrieval-probe.md)
reduced one source-truth run from a broad average of 3,083,245 tokens to 169,104
through runtime-bounded evidence delivery, yet the verifier still missed the
explicit candidate/evidence contradiction. Retrieval solved an attention-cost
problem but not the claim-coverage problem. The next transformation is an
addressable claim audit, not a larger prompt or another retrieval layer.

These observations recur beyond cognition: large code review, structural
refactoring, documentation migration, long-running work, and high-risk effect
execution all need a way to turn task shapes outside a profile's envelope into
work that an agent can settle and a later actor can reconstruct. The intended
harness target is not a frontier model by default. It is a Flash-class,
economical, replaceable daily execution profile whose stable primitives can be
understood, composed, and escalated deliberately.

## Form and name

Create an independently installable core Skill named `task-shaping`. Its
operative definition is: **relative to a conservative Flash-class reference
execution profile, classify requested operations as reliable, guarded,
transformable, or unsupported, then choose the smallest form whose global
obligations can be reconstructed and accepted externally.**

The Skill carries a conservative, revisable task-primitive seed. It does not
declare those primitives universally reliable: `model-evaluation` supplies
profile-specific repeated evidence and human or host admission. `task-shaping`
consumes that evidence to improve harness transformations. Stronger models are
explicit escalation targets rather than hidden assumptions that inflate the
ordinary primitive set.

The name describes the relation-changing action. Rejected alternatives were:

| Candidate | Reason rejected |
|---|---|
| `capability-envelope` | names the diagnostic object but hides transformation and reconstruction |
| `task-decomposition` | reduces the method to splitting even when direct execution, externalized state, bounded retrieval, effect separation, or escalation is the truthful move |
| `agent-task-engineering` | too broad; suggests ownership of planning, tools, execution, and optimization |
| `harness-engineering` | confuses a portable method with the runtime and reintroduces the retired universal-harness surface |

Disconfirm the name if ordinary users repeatedly invoke it for generic planning
or cannot distinguish shaping from orchestration after reading the operative
definition.

## Expression team

- **Primary P15:** select only the minimum task transformation that addresses
  the observed instability while preserving original hard constraints.
- **Supporting P05:** make the envelope and transformation specific to the
  actual execution profile, task history, and relation structure.
- **Supporting P08:** require a disconfirming observation and behavioral
  comparison before claiming improved stability.
- **Supporting P04:** identify the principal task/profile mismatch instead of
  responding to every risk with more instructions or smaller chunks.

The current task lead is P04 because retrieval delivery was the prior suspected
owner and the bounded-expansion result exposed claim coverage as the remaining
principal instability. No new Sequence principle is needed.

## Ownership boundary

| Concern | Owner |
|---|---|
| reusable capability claim about model/provider/harness/profile | `model-evaluation` |
| conservative task-primitive map and semantic transformation relative to the reference envelope | `task-shaping` |
| domain meaning and domain-specific packet boundaries | owning domain Skill |
| information delivery to a prepared unit | `context-engineering` |
| execution queue, concurrency, retries, provider, and persistence | caller or orchestration runtime |
| workload and resource projection | `work-estimation` and authorized resource owner |
| semantic verification and acceptance | designated verifier and human or host authority |

`task-shaping` may return direct execution, prepared units, a discovery need, or
escalation. It does not launch children or create a mandatory planning phase.
When a domain Skill already owns a stable semantic partition, this Skill owns
only an evidenced envelope mismatch not handled there.

`discovery-needed` is an evidence status, not a fifth execution disposition. If
the named profile lacks comparable evidence, use `unsupported-escalate` while
routing the missing reusable capability claim to `model-evaluation`.

The method may use a differential analogy in its conditional concepts layer:
the reference profile solves locally bounded changes, unit granularity acts as
an adaptive step size, and reconstruction integrates local results under source
state, boundary conditions, order, coupling, accumulated error, and global
invariants. The analogy does not rename contract fields or imply that every
task is smooth or decomposable. Failure of this integrability gate is a reason
to merge, order, represent differently, or escalate—not to create more Cells.

## First-slice acceptance

The first slice contains one concise `SKILL.md`, conditional doctrine,
task-primitive, transformation, and evaluation references, generated interface
metadata, and a generated standalone Sequence snapshot. It is published in the
repository skill indexes without adding runtime code. The initial primitive map
contains hypotheses and guarded candidates, not unsupported claims of direct
reliability.

Behavioral acceptance requires:

1. an action task with broad claim-verification pressure produces addressable
   obligations, locally verifiable units, and a bounded reconstruction path;
2. a coherent one-step boundary task remains direct;
3. neither result chooses provider, concurrency, budget, execution, or semantic
   acceptance;
4. conditional references load only when their vocabulary or transformation
   detail changes the decision; and
5. any improvement claim remains proposed until a matched behavioral comparison
   on a separate retained case supports it.

`reliable` is always relative to an explicit task-specific tolerance and
failure consequence. A proposed schema, verifier, repetition, or abstention
rule remains a treatment hypothesis until matched evidence shows that it
detects or contains the known error. Missing tolerance or a known material
residual keeps the operation `guarded`.

Reopen the form if it becomes generic planning, duplicates domain methods,
requires every task to declare an execution profile, cannot keep reconstruction
inside an observed envelope, or adds more coordination than the transformation
removes.

## First-slice evidence

The [first-slice behavioral probe](../../regeneration/evaluations/2026-07-18-task-shaping-first-slice-probe.md)
found and corrected two load-bearing failures: eager reference loading on a
direct task and promotion of historically failing operations by assuming that a
new guard would work. The final development gate kept the matched direct case
direct and classified every semantic operation with known or unresolved error
as guarded. It did not execute the proposed 480-report transformation, so the
primitive seed and reconstruction patterns remain provisional rather than
admitted capability claims.

A held-out mixed code-review case then found a third boundary failure: the Skill
named `code-review` as semantic owner but still instantiated example semantic
packets. The corrected expression stops at envelope, coverage, reconstruction,
and handoff constraints when a domain method owns the partition. Declaring an
owner in prose is not enough after the artifact has already exercised that
owner's authority.
