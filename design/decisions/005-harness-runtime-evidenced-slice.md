# 005 — Harness as Runtime-Evidenced Context Architecture

**Status:** superseded by [decision 026](026-context-engineering-reconstitutes-harness.md)

## Context

The inherited harness correctly identified layered context and cross-session
continuity, but it treated familiar instruction files, hooks, and loading paths
as universal. An isolated forward probe found that it could not demonstrate
that those artifacts reached an agent or improved its action. Its broad prompt
trigger also confused durable context architecture with one-off instruction
drafting.

## Decision

Regenerate harness as the second approved agent-engineering slice. It owns one
gate: whether the actual runtime can deliver the smallest durable context that
enables the project's next judgment.

Its expression team is Primary P09 with P10, P12, and P08 supporting. The
skill begins with a runtime capability map, then assigns L1/L2/L3 from observed
loading behavior rather than from universal filenames. It exposes a compact
human review skeleton and verifies one action task, one boundary task, and the
context path. It bundles a read-only standalone Sequence snapshot containing
the complete one-line Sequence and its selected interpretations.

## Consequences

- A file is L1 only if this runtime demonstrably loads it before the decision it
  is intended to change.
- Unknown tool behavior remains unknown until observed or documented; generic
  hook mechanics become conditional rather than prescribed.
- One-off prompt wording is explicitly outside the harness boundary.
- Configuration validity alone is no longer treated as proof of agent behavior.

## Human approval

Approved through the instruction to first regenerate harness and agent-
engineering skills, and the subsequent requirement that installed skills remain
self-contained, on 2026-07-09.
