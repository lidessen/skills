# 015 — Human-Initiated Formal Operations and Founding Transition

**Status:** approved
**Date:** 2026-07-10
**Approved by:** principal, 2026-07-10

## Context

The collection has an accepted semantic root, specialized methods, bounded Work
Cell execution, and a verified-for-approval organization contract. It can form
a temporary working team around a real contradiction, but it lacks a durable
way to take a bounded mission from a human-authorized agent session through
isolated Git work, independent review, and a human merge.

The operational audit found one remote `main` branch and a history of merged
pull requests, but no branch protection, repository workflow, or pull-request
template. The current checkout is also a large uncommitted transition rather
than a clean integration baseline. The earlier organization audit already
identified that uncommitted layer as a material carrier gap; see its
[historical model](../organization/sessions/artifact-organization-v1/MODEL.md#活动残留).

The question is therefore not how to create a permanent agent government. It
is how a human can deliberately start bounded collective work now, retain
authority through planning, execution, verification, and merge, and later add
new trigger surfaces without granting those surfaces decision authority.

## Form decision

The strongest unchanged case is to keep using ad hoc agent sessions and the
existing occasional PR pattern. It remains adequate for a one-off local edit,
but it cannot reliably distinguish a founding baseline from a mission, preserve
which evidence authorized a merge, or give a later agent a safe worktree and
self-correction route.

The selected minimum form is a stack, not a new runtime:

| Need | Form and owner | Why it is needed | It does not become |
|---|---|---|---|
| Repeatable human-started collaboration | [Operating protocol](../operations/OPERATING-PROTOCOL.md) | Names roles, handoffs, Git boundaries, and correction signals across sessions | a scheduler or permanent agent |
| One founding migration | [Bounded preparation campaign](../organization/sessions/2026-07-10-formal-operations-preparation.md) | Separates the present dirty transition from normal operations and has explicit exit gates | a permanent headquarters |
| Reviewable shared integration | GitHub PR template and verification workflow | Makes source, acceptance, checks, and human decision visible at the merge boundary | a fact source or automatic merger |
| Branch-rule activation | [Deferred settings record](../operations/GITHUB-SETTINGS.md) | States a reversible, evidence-gated external configuration change | an agent permission grant |

No new skill is selected. The repeated judgment is already distributed among
the existing strategic, estimation, execution, verification, and corrective
methods. A central planning runtime, task board, autonomous queue, background
poller, or automated merger would collapse their authority boundaries.

## Trigger boundary

The only active trigger is a **human-authorized agent session**: the principal
asks Codex or another capable agent to work in this repository. The agent reads
the project harness, accepted design, and only the skills relevant to the live
contradiction; it then forms a temporary team under the organization contract.

Future triggers, including scheduled runs and GitHub Issue or event adapters,
are only potential **intake adapters**. They may create a proposed observation
or request for human attention. They may not approve a Strategy Case, allocate
or extend an envelope, accept semantic work, merge a PR, alter `main`, or amend
the Sequence. Each future adapter needs a separate form decision, source and
idempotency contract, and a shadow-run probe before activation.

## Operating formation

`principles/COUNCIL.md` remains the governance projection of P-IDs. It is not
this operating formation. The formal operating team is a set of **standing
seats**, not resident agents or a second semantic committee. A seat is occupied
only for a concrete mission and may be held by a human, an agent, or a
mechanical check where the authority allows it.

| Seat | Activated by | Produces | Cannot do |
|---|---|---|---|
| Principal | human mandate | scope, approval, budget/continuation and merge decision | delegate durable authority implicitly |
| Session convener | authorized agent session | source-bearing mission brief and selected temporary formation | self-approve the result or create a standing queue |
| Planning and articulation advisers | a settled phase, naming decision, or material option | proposed Strategy Case, Work Estimate, or named definition | commit missions, resources, or names as doctrine |
| Execution owner | approved short mission with acceptance and bounds | branch-local implementation and retained execution evidence | accept its own semantic result or merge |
| Independent verifier | a proposed change and named acceptance | mechanical check result and semantic review observation | redefine scope or commit accepted truth alone |
| Integration steward | approved review and human merge decision | clean PR disposition, merge or return-to-work evidence | treat CI green as human approval |
| Correction formation | traceable failure, drift, or settled phase | smallest routed practice, carrier, expression, or principle inquiry | automatically amend a skill, protocol, or Sequence |

This is the operating equivalent of a cabinet, not a hierarchy: planning,
execution, verification, and commitment remain separate under P11 and P13.

## Git and GitHub operating boundary

`main` is the accepted integration baseline, never a scratch area. A shared or
durable mission receives one branch and one isolated worktree. The branch names
the mission; the worktree names the branch location; the PR is the proposed
integration record; the merged commit is the repository's accepted code and
document state. Work Cell traces, generated views, and CI results remain
evidence or projections unless a reviewer promotes a specific record.

The first operational slice provides a PR template and a read-only workflow.
Branch protection is deliberately deferred until a shadow PR proves that the
checks and solo-maintainer merge path are usable. No agent receives push,
merge, auto-merge, issue-closing, or configuration-changing authority from this
decision.

## Founding transition

The current dirty checkout is not retroactively a normal mission. It is the
object of the bounded preparation campaign and must be divided into reviewable
founding tranches before it can become the new baseline. Formal operation starts
only when the campaign's exit gates have passed: a first coherent baseline PR,
two shadow missions using isolated worktrees and the PR template, and a human
decision on branch protection.

The project name is also a founding decision, not an accidental consequence of
the repository directory. The preparation campaign must use
`naming-and-articulation` to produce a small naming record before any public
rename; until then, `skills` remains a descriptive repository handle.

## Sequence expression

- **P04:** the integration-baseline contradiction selects this transition;
  ordinary work does not need its full ceremony.
- **P11 / P13:** proposal, execution, verification, merge, and fact admission
  have named, separate holders.
- **P12 / P14:** durable operating records survive in declared sources, while
  PR views, CI results, and generated traces remain derived evidence.
- **P15 / P16:** a protocol, one campaign, and thin GitHub carriers enable the
  next action without inventing a governing runtime.

## Acceptance

- The active trigger is a human-authorized agent session; event and schedule
  triggers are explicitly non-authoritative future adapters.
- A reader can distinguish the principle council from the operating seats and
  identify the sole human merge/commit authority.
- The protocol gives one mission an inspectable route through worktree, PR,
  verification, disposition, and correction.
- The current dirty transition has a bounded founding campaign and does not
  silently become a release.
- The repository contains a PR template and a non-mutating verification
  workflow; branch protection is applied only after the stated shadow evidence.
