# 017 — Aesthetic Practice by Adaptation and Trial

**Status:** accepted for pilot
**Date:** 2026-07-10
**Pilot authorized by:** principal

## Context

The project needs a durable way to direct illustration, documentation, UI, and
public expression. It must not invent an aesthetic bureaucracy from scratch or
turn visual polish into a false proof of openness, accessibility, or capability.

The available evidence does not establish a Sequence gap; it supports absorbing
proven forms before creating a new method or runtime. The
[U.S. Web Design System design principles](https://designsystem.digital.gov/design-principles/)
provide a user-needs-first evaluative lens; its [maturity model](https://designsystem.digital.gov/maturity-model/)
separates principles, guidance, tokens, components, and contributions. The
[GOV.UK contribution criteria](https://design-system.service.gov.uk/community/contribution-criteria/)
require evidence of usefulness, usability, consistency, and versatility before
promotion. The [Google developer documentation style guide](https://developers.google.com/style)
uses a hierarchy in which project-specific guidance comes first and a general
guide is fallback rather than doctrine.

## Decision

Adopt an **Aesthetic Practice Pilot**. It adapts four mature forms without
copying any source system's visual language:

| Mature form | Adapted project form | Boundary |
|---|---|---|
| Design principles | founding mandate plus a task-specific Aesthetic Case | no global taste rule |
| Creative/art direction brief | Work Cell Expression Brief | no autonomous art director or undeclared tool |
| Editorial hierarchy | project expression guidance first, external editorial guide as fallback | no universal prose/style manual |
| Contribution and review process | studies graduate to directions only after repeated use and human review | no gallery, asset dump, or automatic canon |

The pilot creates `design/aesthetics/` as a source-bearing practice area. It
does not yet create an installable skill, design-token package, component
library, image registry, visual evaluator, or Work Cell runtime capability.

## Pilot formation

The formation is temporary and task-bound:

| Role | Contribution | Cannot do |
|---|---|---|
| Principal | supplies preference seed and accepts a public direction | delegate aesthetic acceptance implicitly |
| Aesthetic facilitator | prepares comparative cases, references, briefs, and study route | impose taste or accept final work |
| Maker / Work Cell | produces a bounded document, UI, or visual artifact under declared tools | claim absent capability or self-approve |
| Reviewer | judges the artifact in its real audience/task context | promote a general direction from one attractive result |

The formation communicates a real capability truthfully. It is not a propaganda
office: it must show uncertainty, constraints, and public-interest boundaries
rather than manufacture consent.

## Work Cell boundary

A pilot case lowers into existing Cell DNA, scope, capabilities, acceptance,
and evidence fields. A cell may use only a tool actually available through its
driver/environment; an image or browser task names that capability explicitly.
If the capability is absent, it returns the limitation and a next practice,
not a fabricated visual result. Runtime extension is reconsidered only after a
real case proves that this contract is insufficient.

## Graduation rule

A direction, editorial pattern, UI pattern, or visual token enters the shared
area only when it has appeared in at least two independent real cases, has a
named use/contrast, retains attributed references and accessibility/production
constraints, and is accepted by the Principal or named reviewer. A single seed
or generated image remains a provisional preference signal.

## Acceptance

- A project aesthetic practice area distinguishes seed, case, study, direction,
  and reference without creating a second design source.
- The first seed is explicitly labeled as the Principal's preference, not an
  objective or universal aesthetic truth.
- One document/UI and one visual-asset case test the seed under real tasks;
  one superficial-polish task declines the practice.
- A future `aesthetic-cultivation` skill is considered only if these cases show
  a repeated agent-facing judgment that project-local references do not teach.
