# Project Cognition Form Decision

**Status:** approved
**Date:** 2026-07-16
**Approved by:** principal
**Host design:** [Project cognition bootstrap Strategy Case](2026-07-16-project-cognition-bootstrap-strategy-case.md)
**Evidence:** [Shilu bootstrap and refresh probe](../../../regeneration/evaluations/2026-07-16-project-cognition-bootstrap-and-refresh.md)

## Concrete need

- **Actor and next decision:** an agent responsible for a substantial project
  task must decide how to decompose the named question, what project relations
  each bounded Cell must recover, how the local records reconnect, and which
  prior cognition facets a source change invalidates.
- **Observed action gap and evidence:** Work Cell and Swarm can execute prepared
  Cells, but deliberately do not own semantic decomposition, synthesis, or
  refresh. In the Shilu probe, a caller had to invent those judgments across
  bootstrap, local repartition, withheld evaluation, and delta refresh. The
  judgments recurred and changed the result: the partitioned carrier recovered
  material authority, partial-failure, parity, and recovery relations missed by
  the whole-project baseline, while the delta required a different partition.
- **What must survive:** the repeatable agent judgment, its trigger and
  boundaries, and a target project's small source-linked cognition index and
  semantic facets when later tasks benefit from them. Raw run records can remain
  cold evidence. Model prompts, temporary packet plans, and unsuccessful output
  protocols need not become permanent doctrine.
- **Verifier and committer:** a source-aware reviewer verifies cognition claims
  against the named project revision and accepted design. The target project's
  authorized human or mainline process decides whether to retain a projection.
  A Cell, Swarm, Skill, or projection cannot accept its own semantic result.
- **Strongest keep-as-is case:** substantial tasks can keep building temporary
  local models directly from source. This avoids a new Skill and stale
  projections, and remains the right path for one-off questions or small
  projects. It fails when later agents must repeatedly reconstruct the same
  responsibility, authority, causal, state, and change relations or cannot
  selectively refresh prior understanding.

## Principal form contradiction

The recurring gap is a missing **agent judgment**, not a missing execution
engine: the actor must adapt decomposition and refresh to the actual project,
but the reusable mechanics intentionally refuse to encode domain partitions.
Leaving the method only in experiment reports makes every caller reconstruct
it. Moving it into Work Cell or Swarm would make generic infrastructure own
project semantics and would freeze provisional facet names into a taxonomy.

## Form comparison

| Candidate | Decision changed | Authority/source effect | Disposition |
|---|---|---|---|
| Keep temporary task models only | none; every caller invents its own decomposition and refresh rule | source remains clear, but no reusable judgment or cross-session carrier | retain for bounded one-off questions; insufficient as the only form after repeated bootstrap and refresh evidence |
| Local instruction or reference | gives one repository a suggested procedure | risks mixing local facts with portable method and is not independently installable | reject as primary form; target projects may add a thin local instruction only to declare their chosen source and projection path |
| Installable domain Skill | lets an agent decide whether cognition is needed, form a task-specific partition, synthesize source-linked facets, evaluate them, and select refresh scope | Skill owns method only; target source remains authoritative and generated cognition remains projection | **select** |
| Rebuildable project-local projection | lets later agents recover accepted working understanding without replaying every run | small index and semantic facets are derived from named revisions; hashes and anchors expose reconstruction and invalidation | **select conditionally as Skill output**, only when reuse justifies persistence |
| Local source/ledger script | can mechanize hashes, source inventory, changed-path collection, or consistency checks | would be a replaceable adapter over declared sources, not a cognition source | defer until repeated manual error proves a mechanical gap; do not create in the first transition |
| Cognition runtime or Work Cell carrier | could enforce storage, invalidation, retrieval, or refresh transitions | would introduce state and lifecycle authority not required by the evidence and duplicate existing execution records | reject now |
| Shilu protocol or backend | could later store governed Entries and projections for a project that already adopts Shilu | would make this project depend on another system and could confuse Shilu Entry authority with derived cognition | reject as a dependency; permit only as a future optional adapter after a separate form decision |

## Selected form stack

- **Form and owner:** an independently installable domain Skill, provisionally
  `project-cognition`, owns the repeated agent judgment. When persistence changes
  a later action, it may produce a small project-local projection consisting of
  a routing index and meaning-owned facets.
- **Content owner and unresolved domain question:** `skill-engineering` should
  express and behavior-test the Skill. The new domain owner must still specify
  the minimum method for deciding whether cognition is warranted, choosing a
  task-specific semantic partition, source-checking synthesis, reconnecting
  boundaries, and selecting delta invalidation. This record does not define
  those instructions.
- **Source and projection boundary:** the target repository's named source
  revision, accepted design, tests, and governed records remain authoritative.
  Cognition files are rebuildable access projections. Work Cell run records are
  execution evidence, not project facts; summaries and indexes are projections
  over those records and the named project source.
- **Execution boundary:** Work Cell guarantees a bounded executable Cell and
  its artifacts, checks, usage, and run record. Swarm/orchestration guarantees
  release, bounded concurrency, isolation, and settlement. Neither contains
  fixed `authority`, `lifecycle`, `review`, repository, or project-cognition
  packet types.
- **Selection boundary:** the activated agent inspects the actual project,
  question, source relations, and change field, then chooses this activation's
  partition. A Skill may offer candidate cuts and failure/repartition questions,
  but no preset is mandatory. The same project may use different partitions for
  bootstrap, review, refactoring, documentation, or delta refresh.
- **Context boundary:** `context-engineering` may deliver the selected source or
  cognition slice through the actual agent runtime. It does not author the
  cognition method, choose partitions, or confer fact authority.
- **Size boundary:** use a small index and one coherent semantic question per
  facet. File size is soft agent-readability guidance. Split only when one agent
  can no longer read and revise the question coherently; do not add byte-based
  chunks, nested manifests, or hard report limits by default.
- **Non-scope:** no database, vector index, daemon, universal ontology, fixed
  decomposition taxonomy, automatic fact admission, mandatory project setup,
  or Work Cell schema expansion belongs to the first form.

## Lifecycle and use boundary

The Skill should decline persistence for a one-off bounded question. A durable
projection is warranted only when named later actors or decisions will reuse
it. Bootstrap names its source revision; refresh follows material changed
relations rather than a timer; unchanged facets may be retained only with an
explicit source-backed reason; and a later task may bypass or rebuild the
projection when direct source reading is cheaper or safer.

## Form probes

| Probe | Supported observation now | Required acceptance observation for the selected form |
|---|---|---|
| Action | the Shilu caller successfully changed decomposition between bootstrap, failed-packet recovery, and delta refresh | an agent using only the installed Skill plus a named project can choose a bounded partition, produce a source-linked carrier, and answer a withheld downstream question better than an unstructured temporary model |
| Boundary | generic Work Cell and Swarm completed execution without needing project facet vocabulary | the Skill declines a small one-off question, routes ordinary code-review method to `code-review`, and does not add domain fields to Work Cell or orchestration |
| Authority | the probe source-checked claims and kept recoverable protocol-error artifacts as proposals | the projection identifies its source revision and verifier, cannot mark itself accepted, and can be deleted and rebuilt without loss of authoritative state |
| Context | the index and question-owned facets avoided loading all raw Cell records | an activated agent loads only the index and relevant facets/source anchors, while volatile target-project facts stay out of the portable Skill |
| Handoff | this decision names `skill-engineering` and stops before implementation | the next Mission owns domain-method expression and tests without reopening runtime design by default |

## Minimum transition

- **Smallest next change:** after Principal approval, open one bounded
  `skill-engineering` Mission for the domain Skill. Reuse the proven Shilu cases
  as behavior fixtures; do not add a script or runtime in that first slice.
- **Acceptance observation:** an external or clean target context can install the
  Skill, dynamically choose a project-specific decomposition, persist only a
  justified small projection, selectively refresh a bounded delta, and preserve
  source/verification authority without knowing Work Cell's internal JSON.
- **Disconfirming observation:** the Skill merely repeats `code-review` or
  `context-engineering`; agents still need undocumented caller knowledge;
  dynamic partitioning cannot be expressed without runtime domain state; most
  small changes invalidate the whole projection; or maintaining the carrier is
  consistently more expensive than rebuilding the required source model.
- **Residual uncertainty:** only one second project and one delta have been
  tested. The first Skill probe must include a structurally different project or
  a deliberately different task type before claiming a general method.
