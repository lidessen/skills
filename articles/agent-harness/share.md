## 一、为什么Agent上下文天然是对话

### 1.1 LLM：根据上下文作最有可能的续写

LLM在生成回复的时候，其实没有所谓的记忆、不存在所谓的我, 回答、推理、调用工具,都是对续写一段文本的尝试。

**所以让 LLM 做事,其实是创建上下文引导**。

**图 1.1 预测循环**

```
  输入:  "今天天气很"
             │
             ▼
        ┌────────┐
        │  LLM   │
        └────────┘
             │
             ▼
  输出:       "好"
             │
             ▼
  追加:  "今天天气很好"
             │
             ▼
        ┌────────┐
        │  LLM   │
        └────────┘
             │
             ▼
  输出:       ","
          ...
```

### 1.2 为什么对话是天然上下文

首先GPT最早的时候就是以对话的形式推出的，并且直至今天我们仍然未跳出对话的框架。

一个不负责任的推测，对话是一种交互相对比较自然的滚动式上下文。

**图 1.2 消息序列追加**

```
  第 1 轮喂给模型:
    [user] 帮我读 main.py
    → 预测

  第 2 轮喂给模型:
    [user] 帮我读 main.py
    [assistant] <read_file main.py>
    [tool_result] (文件内容)
    → 预测

  第 3 轮喂给模型:
    [user] 帮我读 main.py
    [assistant] <read_file main.py>
    [tool_result] (文件内容)
    [assistant] 这个文件做了...
    → 预测

  每轮都把整段历史喂回去——追加一条,再整段过一遍。
```

### 1.3 和世界交互

如今的Agent对话上下文，说是对话其实不准确，LLM在coding场景的深耕，其实已经往对话历史中引入了大量对话之外的“噪声”，但其实不是噪声，应该说是非直接对话上下文，比如工具调用就是一个很典型的形式，给原本只有对话才能推动滚动的上下文引入了新的推动来源。一个agent自己产生自己消费的内循环机制，如果把对话比作外循环的话。

其实把对话和工具调用各算一条循环,只是一个开始。往后推,上下文里完全可以有更多条独立脉络并存—— 多个 worker 并行处理子任务、agent 之间的异步协作、跨会话的长期观察、定时触发的复盘,每一条都有自己的节奏和产物。只不过到今天为止,所有这些东西都被挤进了同一条扁平的对话序列,像几个独立模块的代码被塞进一个 god file。

**图 1.3 Agent 循环**

```
  用户 prompt
      │
      ▼
  ┌──────────────┐
  │  组装上下文  │ ◀───────────────┐
  └──────────────┘                 │
      │                            │
      ▼                            │
  ┌──────────────┐                 │
  │   调用模型   │                 │
  └──────────────┘                 │
      │                            │
   ┌──┴──┐                         │
   │ 分支 │                        │
   └──┬──┘                         │
      ├── 输出文本 ──→ 返回用户    │
      │                            │
      └── 输出 tool call           │
              │                    │
              ▼                    │
          执行工具                 │
              │                    │
              ▼                    │
          结果追加回上下文 ────────┘
```

### 1.4 Claude Code 怎么组装上下文

真正让这个循环跑起来,问题其实立刻就来了:几十个文件要让模型知道、工具返回几万行搜索结果要塞回上下文、上下文跑久早期内容越堆越多。

Claude Code 的答案是四项机制:

- **Skill**:描述常驻,正文按需加载
- **Tool 定义**:列表常驻,结果裁剪
- **Subagent**:独立上下文干脏活,只把结论返回主会话
- **Compaction**:高水位自动压缩早期对话

**图 1.4 上下文窗口组成**

```
  ┌─────────────────────────────────────┐
  │  System Prompt                      │
  ├─────────────────────────────────────┤
  │  CLAUDE.md  (项目指令)              │
  ├─────────────────────────────────────┤
  │  MEMORY.md 前 200 行  (auto memory) │
  ├─────────────────────────────────────┤
  │  Skill 描述列表  (正文按需加载)     │
  ├─────────────────────────────────────┤
  │  Tool 定义                          │
  ├─────────────────────────────────────┤
  │  对话历史  (可能被 compaction)      │
  ├─────────────────────────────────────┤
  │  最新用户输入                       │
  └─────────────────────────────────────┘
                    ▲
             上下文窗口上限
```

把四件事放在一起看,**Claude Code 的工程重点其实不在模型,而在上下文组装本身**。每次调用模型前,系统都在判断:什么该留在上下文里、什么不该、什么该压缩、什么该外包出去。

**harness 的本质,其实是一台上下文组装机**。Codex、Hermes、Agents SDK 做的都是同一件事。**Agent 不是一段 prompt,是一套围绕上下文做信息管理的运行时系统**,这一点算是行业共识。

### 1.5 上下文并不能无限增长

> **[AI 图 1.5:对话膨胀对比]**
两幅并排的上下文窗口可视化。左侧「短对话」:3-5 轮消息,每条用高饱和色块标示,信息密度高、占比清晰。右侧「30 轮后」:同等大小的窗口,绝大部分被灰色块(礼貌措辞 / 过程性回复 / 工具原始输出)占满,中间零星几个高饱和色块(核心状态信息)被稀释包围,占比约 5%。关键是用颜色和比例表达「稀释感」——ASCII 做不出这层对比。
> 

问题一目了然。

**真正有效的状态信息其实很少**。30 轮对话里,对后续推进有价值的核心事实就那么几条——几千 token 换几行有效信息。

**根源其实是多条本该独立的线,被挤进了同一条上下文**。一条模型回复同时在承担好几件事——给用户看的表达、工具调用的发起、下一轮预测的输入、任务进展的状态记录。每一件事本该各自成线,有自己的节奏和输出要求;被压在同一段上下文里,互相稀释。外循环(对话)和内循环(工具调用、推理)的产物混在一起,只是这个问题最显眼的一个表现。

短任务不明显。任务一拉长,上下文快速膨胀,膨胀的大部分其实不是有效信息。

**追问一句"Agent 能不能跨三天、跨多个话题地持续运行"——到这一步,把所有东西都塞进一条上下文的做法,其实已经走到了自己的边界**。

## 二、上下文只能是对话吗？

### 2.1 长对话能有多长？

设想一个场景:让 Agent 维护一个中型项目三天。第一天排查 bug + 加新功能,第二天发现 bug 没修干净回到老话题 + 提新重构,第三天把所有改动整理成一份 PR 说明。

真实工作里普通的三天。但在当前对话模式下非常难实现:

- **放同一个对话里**:上下文第一天下午就膨胀严重,第三天基本不可用
- **开新对话**:模型每次都是白纸,每次得从头讲一遍——可自己都不一定记得全

### 2.2 上下文破破烂烂，Harness 缝缝补补

**图 2.2 现有方案对比**

```
  方案                  作用域      本质                局限
  ────────────────────────────────────────────────────────────────
  Compaction            对话内      有损压缩历史        细节丢失
  持久化 transcript     跨对话      按需回查            不持续在场
  CLAUDE.md             会话起点    手写起手素材        需要维护
  Auto memory           会话起点    自动积累起手素材    只在起点加载
  24h Synthesis         跨对话      合成关键洞察        粒度粗

  共同前提:「对话是上下文主体」,只是让对话跑得更久。
```

主流 harness 都意识到了这个问题,各自给出了答案。把这些答案放在一起看,差异其实小得可疑——它们都共享一个前提:**对话是上下文主体,要做的是让对话能跑得更久**。

memory 是对话的起手素材,compaction 是对话的内部压缩,历史搜索是对话之外的回查通道。所有方案都没有动一个底层设定:**上下文 = 对话**。说得更准确一点,这些方案**都没有真正区分外循环(对话)和内循环(agent 自产自消的那部分)**——补丁基本都打在外循环上,内循环的问题其实一直没被单独处理。

补丁打了这么多,对话还是那个对话。

要真正解决,其实得换一个思路:**不要让对话当上下文主体**。

### 2.3 续写什么不是续写

> **[AI 图 2.3:对话模式 vs 事件流模式]**
两个同等大小的上下文窗口容器并排对比。左侧「对话模式」:内部密集排列 user / assistant / tool 往返消息,颜色交织、消息拥挤、看起来填满且压抑。右侧「事件流模式」:内部是稀疏的几条结构化事件(系统通知 / 任务派发 / 完成报告),每条独立成块、之间有明显呼吸空间,信息密度高但视觉留白多。核心对比:同样的窗口容量,两种结构的「装载效率」天差地别。
> 

把上下文从对话换成**工作区信息流**:项目当前正在发生什么,以客观记录的形式排成一串——系统通知、任务派发、完成报告。每一条都是高密度信息,没有礼貌措辞、没有过程性表达、没有工具调用原始输出。

其实这个思路的本质,是**把原本挤在一条对话里的多条脉络,各自单独拎出来成线**。对话还在,但它从"全部上下文"降格为"其中一条脉络"。每个任务由独立对话式 agent(worker)完整执行,干完之后对话流被压成一条几十字的"完成报告",作为事件写入上一层;与此同时,其他脉络——跨会话的长期观察、agent 之间的异步协作、定时触发的复盘——也以同样的格式写进来,各自独立,互不干扰。

原来挤在一条 god file 里的东西,第一次有了拆开的空间。

### 2.4 上下文的层级化

> **[AI 图 2.4:层级化框架(封面级)]**
全书核心概念图,纵向三层结构,从下到上 L0 → L1 → L2:
> 
> - 底层 **L0**:4-5 个并排的对话窗口,每个窗口塞满密集的 user/assistant/tool 消息,标签「对话式 worker · 短生命周期 · 任务结束即销毁」
> - 中层 **L1**:单个上下文窗口,内部是稀疏的结构化事件条目(系统通知 / 任务派发 / 完成报告),标签「事件流 · 任务级积累」
> - 顶层 **L2**:更窄的窗口,内部仅几条高度抽象的摘要事件,标签「阶段总结 · 周/月级」
> - 层间箭头:向下「派发任务 + 投影事件」,向上「结构化完成报告」
> - 右侧辅助标注:L0 = 工程师 / L1 = 部门经理 / L2 = 总监
> 
> 风格简洁专业,适合全书封面或首页插图。重点在于让读者一眼看明白「每往上一层,信息密度更高、积累更慢、时间跨度更长」。
> 

**图 2.4 · 简化版 ASCII**(作为正文里的快速参考)

```
  ┌───────────────────────────────────────┐
  │ L2: 阶段总结事件流                    │
  │     (周/月级,非常稀)                │
  └───────────────────────────────────────┘
                ▲ 汇报
  ┌───────────────────────────────────────┐
  │ L1: 任务事件流                        │
  │     (任务级,几十条事件)              │
  └───────────────────────────────────────┘
         ▲ 报告           │ 派发
         │                ▼
  ┌──────────┐ ┌──────────┐ ┌──────────┐
  │ L0 对话  │ │ L0 对话  │ │ L0 对话  │
  │ worker   │ │ worker   │ │ worker   │
  │ (短生命) │ │ (短生命) │ │ (短生命) │
  └──────────┘ └──────────┘ └──────────┘
```

从最底层编号:

- **L0**:对话式 worker,执行具体任务,任务结束即销毁
- **L1**:统筹层,上下文是事件流,每条事件是 L0 的完成报告
- **L2 及以上**:按场景需要,更高层把下层事件流进一步抽象

逻辑类似公司管理结构。一线工程师感知粒度是"这个函数的这个 bug",部门经理感知粒度是"这个模块这周的进展",总监感知粒度是"这个产品线这个季度的方向"。**总监不应该知道每个 ticket 的细节,他该知道的是被压缩过的高层事件**。

以上说的都是**纵向分层**。其实还有另一个维度: **同一层里可以并存多条独立脉络**——也就是横向的分模块。L0 层从来都不是一条对话,同一时刻可以有几十个 worker 在各自的上下文里跑,互不干扰;L1 层也一样,同一条事件流可以被多个订阅方共用,每个订阅方只看自己关心的事件类型,各自跑各自的处理循环。

这样一来,**上下文的架构,其实和软件架构是一回事**——纵向分层决定信息密度,横向分模块决定职责边界。对话式 agent 的上下文之所以撑不住,本质上是一份没有结构的 god file;事件流架构做的事情,就是把它拆成该拆的层、该拆的模块。

每层只装它这一层该看的东西。往上每升一层,信息密度更高、积累速度更慢、时间跨度更长。

### 2.5 捡起西瓜丢芝麻

**图 2.5 三天时间轴**

```
              Day 1               Day 2               Day 3
            ──────────          ──────────          ──────────

  L1 事件流: [bug]─[功能]       [再排查]─[重构]     [整理 PR]
             ─────────────────────────────────────────────────→
               │    │              │    │              │
               ▼    ▼              ▼    ▼              ▼
  L0 对话:   [w01] [w02]         [w03] [w04]         [w05]
             结束销毁            结束销毁            结束销毁

  L1 持续存在 · L0 不断生生灭灭
```

分层结构下:L0 有几十个短生命周期的对话,完成后销毁、只留报告;L1 有一个长生命周期的事件流,积累三天所有任务报告。第三天派它"整理 PR 说明"时,L1 不需要回忆——上下文里就是三天的事件流,每条都是高密度报告。

**上下文不再膨胀**(三天几十条事件,对比几十万字原始对话流)。**对话质量不再退化**(每个 L0 都是短的、聚焦的、新鲜的)。

现有 harness 让对话同时干"任务执行"和"长期积累",所以两边都不够好。分到不同层各干各的,两边都回到舒服的工作区间。

## 三、重新设计上下文

**Q1. 事件长什么样?**
一次**状态变化的最小可陈述单元**。判断标准只有一条:离开上下文后这条事件是否还能独立被理解。外部世界(webhook、告警、用户消息)也产生事件,以同样格式写进流。

**Q2. worker 怎么被派发?**
L1 给 worker 两样东西:一句话的明确目标 + 从事件流裁出的相关事件子集(不是整条事件流)。worker 不知道 L1 完整事件流存在。**记忆在 L1,不在 worker**。

**Q3. 对话怎么压成事件?**
用独立模型按固定 schema 抽取:做了什么、结果、改动文件、遗留风险、未完成事。**压缩不是摘要,是抽取**。独立调用是为了避开 worker 本身的主观色彩——让它"总结一下自己做了什么",它会倾向性地美化,就像工程师写周报。

**Q4. L1 每轮做什么?**
三种动作:直接响应 / 派新 worker / 向上汇报。不是 if-else 代码,是 L1 基于事件流"预测"出来的。规则加多了,L1 就不再是 agent,是一个用 LLM 当 CPU 的复杂 workflow 引擎。

**Q5. 防泄漏和防丢失?**
泄漏:worker 完成后的压缩是强硬边界,L1 不读原始对话。丢失:压缩 schema 强制包含"遗留风险"和"未完成事"字段——即使 worker 主观认为修复 OK,只要在对话里提过风险,压缩必须填进去。

**最小原型**:复用现成 L0 runtime(Claude Code / Codex / Agents SDK)+ append-only 事件存储(JSONL 或 SQLite)+ L1 循环 + 压缩调用。200-500 行代码能跑。**真正复杂的地方不在机制,在事件 schema 的设计**——这需要对具体业务的理解。

## 四、规模化应用

### 4.1 需要什么样的业务环境

分层 harness 再精巧,单独放出来只是一个空壳。要跑起来,需要一个"高度信息化"的业务环境,两个条件:

1. **业务状态对 agent 可读**(MCP 协议正在做这件事——标准化 tool 和 resource)
2. **业务动作对 agent 可写**(每个业务动作封装成 agent 可调 API)

只读不写,agent 是分析器;只写不读,agent 是傀儡。很多公司 agent 效果不好,根源往往不在 agent 自己,而在业务系统——状态不结构化、动作没有 API、每次调用都要 session 认证。**基础设施不齐,分层 harness 再好也落不下来**。

### 4.2 工位化:agent 应该接管什么

想让 agent "做某个岗位的全部" 是常见错误——"AI 客服" / "AI 工程师" / "AI 运维" 听起来很酷,实际做起来 scope 太大,效果飘忽。

**更实际的做法是工位化**。把岗位拆成一个个工位,每个工位有:

- 明确的输入事件(工单创建、代码提交、告警触发)
- 明确的输出动作(回工单、改代码、升级告警)
- 明确的完成标准(工单关闭、PR 合并、告警消除)

一个普通公司里这样的工位非常多——填一份表、处理一条 L1 工单、跑周期性报表、版本号升级。单独看不够成一个岗位,加起来占了大量人力。

副作用:工位定义清楚,事件流的结构就自然定下来了。**事件 schema 不是从架构角度设计出来的,是从业务工位自下而上长出来的**。

### 4.3 数字流水线

前面讲的横向分模块,放到业务场景里具体长什么样?一个客户投诉处理流水线就是典型例子。

**图 4.3 客户投诉处理流水线**

```
  事件流 (时间轴):
    客户消息 ▸ 工单创建 ▸ 工单分配 ▸ 工单处理中 ▸ 工单已解决 ▸ 客户回复

  ┌───────────────────────────────────────────────────────────┐
  │  agent            订阅事件         产生事件                │
  ├───────────────────────────────────────────────────────────┤
  │  客服工位         客户消息         工单创建                │
  │  分类 agent       工单创建         工单分配                │
  │  处理工位         工单分配         工单处理中 / 已解决     │
  │  客服工位         工单已解决       客户回复                │
  │  复盘 agent       (每日全扫)      (生成报告 · 不写流)    │
  └───────────────────────────────────────────────────────────┘

  每个 agent 只订阅自己关心的事件类型,产生新事件。
  中央没有调度器 —— 事件流本身就是调度。
```

**五个 agent 之间没有一句对话**,完全通过事件流协作。每个 agent 是一条独立的脉络,各自订阅自己关心的事件、产出自己关心的事件,彼此没有直接通信,也不需要——这就是 2.4 讲的横向分模块,在业务里的具体形态。

### 4.4 小团队的位置

基础设施层(模型、通用 harness、记忆系统)大厂在做,会越做越好。真正的价值在**业务层**:某个行业的工位怎么拆、事件 schema 是什么、敏感度档位怎么定、哪些工位适合 agent 哪些不适合。

这些问题,通用方案答不上来,需要对行业的深度理解——客服行业和 devops 行业需要的工位完全不同,同样是客服,SaaS 和电商的事件 schema 也不一样。

**不做通用 agent,做"这个行业的数字流水线"**。通用层交给大厂,业务层靠领域内的人自己长出来。分层 harness 只是一个承载容器,真正决定效果的是容器里装的是什么。

## 写在最后

LLM 是预测机器,它基于一段文本预测下一段。上层系统的所有设计,都在回答"这段文本应该是什么"。

对话是这段文本的第一种形态。但任务一长、脉络一多,这个形态就撑不住了。再往下看,其实是**给上下文加一层架构**——纵向分层让信息按时间尺度落位,横向分模块让不同脉络各走各的线。对话没消失,它被推到最底层 L0 服务于具体任务;跨脉络的长期积累上升到事件流层。

事件流不是发明,只是把原本挤在一条对话里的东西,按它本来的结构排好。真正的门槛不在机制,在**为具体业务设计对应的事件 schema**——机制是通用的,schema 是领域的。

这个方向会把 agent 从陪聊助理变成长期驻守在工位上的执行者,和人组成一条数字流水线。流水线已经存在,它就是 Claude Code、Codex、Hermes 里一块一块长出来的那些机制。**接下来不是再发明什么,是把这些零件拼起来,放到具体业务里,看它能不能真正跑起来**。
