## 一句话核心思想

**当前所有 Agent harness 都把"对话"当作上下文主体,这是对话这个形态在承担不属于它的职责;应该把上下文从对话里抽出来,按信息层级分层组织——对话只做末端的任务执行,长期积累交给更高层的事件流。**

## 核心论点链(必须按这个顺序建立)

1. **LLM 是一台预测机器**。它只会看一段文本预测下一段,没有记忆、没有状态。所有上层系统的形态都由这个事实决定。
2. **Agent 必然长成对话的样子**。因为要让预测机器持续工作,唯一的办法是把新信息不断追加到已有文本后面,整段喂给模型。对话不是界面选择,是 LLM 能承载任务的底层形态。
3. **工具调用只是对话流里的一种新消息类型**。模型输出一段"调用工具 X"的文本,harness 执行后把结果作为新消息追加回对话。Agent 的行动力就是这么来的。
4. **Harness 的本质是上下文组装机**。Claude Code 的所有机制(skill 懒加载、tool 结果裁剪、subagent、compaction)都在回答同一个问题:什么该进上下文、什么时候进、以什么形式进。这是行业共识,Codex、Hermes、Agents SDK 都在做同样的事。
5. **对话式上下文有一个根本局限:信息密度不够、粒度太细**。因为对话里的每条消息同时承担多重职责——既是给用户看的表达,又是工具调用的发起,又是下一轮预测的输入,又是任务状态的记录。这些职责叠在一起,信息被稀释。短任务不明显,长任务问题就放大。
6. **现有 memory / compaction / 持久化 transcript 都是在打补丁**。它们在"对话是上下文主体"这个前提下让对话跑得更久,但没有改变底层设定。
7. **解法是上下文层级化**。把对话推到最底层 L0(只做具体任务执行),上面一层 L1 用事件流(系统通知、任务派发、完成报告)做长期统筹,再往上可以有 L2、L3,按场景需要。每层信息密度更高、积累更慢、时间跨度更长。类比公司管理层——不是 CEO 知道的更多,而是 CEO 不应该知道每个 ticket 的细节。

## 全书四章规划

- **第一章**:讲清楚 Agent harness 的本质,以 Claude Code 为样本。结尾暴露对话式上下文的局限。(已写)
- **第二章**:提出"让 Agent 跑三天"这个真实问题,分析现有方案都在打补丁,提出上下文层级化框架。(已写)
- **第三章**:怎么实现。回答第二章结尾那五个开放问题——事件粒度怎么定、worker 怎么派发、对话怎么压成事件、层间信息流怎么设计等。基于"LLM 根据对话预测后续内容"这个本质,设计的思路是**构造一个工作区信息流,代替对话信息流**。
- **第四章**:怎么工程化应用。

## 写作风格要求(非常重要,不要违反)

- **受众假设是"什么都不懂的小白"**,用朴素语言解释,第一次出现的术语都要说清是什么。
- **少用比喻**。尤其避免"潜艇压载舱""交通灯""厨师"这类华丽比喻。公司管理层的类比可以用,因为它帮助建立直觉而且贴近原意。
- **段落要短**,一段讲一件事。不要一段塞三四个要点。
- **不要用"prompt/context economics""渐进式信息披露"这类抽象术语**,改用大白话:"按需加载""主动挑选"。
- **技术细节只保留对主线有用的**。Claude Code 版本号、StreamingToolExecutor 类名、Hermes 的 9200 行代码、iteration budget 阈值这类佐证性细节一律删掉。
- **说服力来自展示而非论证**。重要的地方用真实形态的 context prompt 抓取给读者看,让读者自己感受到问题,不要光靠描述。

## 已知事实性约束(不要写错)

- **Claude Code 没有 rules**。path-scoped rules 是 Cursor 或其他工具的概念。Claude Code 的 memory 机制是 CLAUDE.md(手写项目指令,每次会话全量加载)+ auto memory / MEMORY.md(Claude 自己在工作中写的笔记,放在 `~/.claude/projects/<project>/memory/`,每次会话加载前 200 行或 25KB,更详细的拆到 topic 文件按需加载)。
- **claude.ai 的跨对话 memory synthesis** 是 claude.ai 产品层面的,不是 Claude Code 的——每 24 小时合成一次跨历史对话的关键洞察。
- **Anthropic API 有独立的 memory tool**,让模型主动管理 `/memories` 目录。
- **Claude Code 的 context 机制包括**:skill(描述常驻、正文按需)、tool 定义(列表常驻、结果裁剪)、subagent(独立上下文)、compaction(高水位压缩)。这四项是第一章讲的,不要加第五项或把名字混淆。
- **行业共识的例子**:OpenAI Codex(官方定义为 harness)、Nous Research 的 Hermes Agent、OpenAI Agents SDK。都是围绕"上下文管理"做的运行时系统。

## 关键措辞约定

- 统一用"对话""对话流""对话式上下文",不要混用"会话""session""聊天"等。
- 统一用"harness",不要翻译成"脚手架""运行时"——harness 是已经立住的术语,全文第一次出现时解释一下即可。
- **L0 / L1 / L2 从底层编号**(L0 是最具体的执行层,数字越大越抽象)。不预设层数,不同场景按需分层。不要用"一级/二级上下文"或"主/从上下文"这类固定层数的说法。
- "事件流"对应高层上下文,"对话流"对应低层上下文。事件的典型形式是"系统通知、任务派发、完成报告"。

## 一个不点破但始终在推进的暗线

从第一章到第二章,作者在反复让读者看到:**业界已经在朝"让上下文更精心构造"这个方向走了**——skill 的懒加载打破了"工具定义全量常驻",subagent 打破了"所有工作在主对话里",compaction 打破了"历史完整保留"。每一步都是在重新回答"上下文应该以什么为单位组织"。层级化是这条轨迹的自然延伸,不是突兀的创新。

读者走到第二章结尾,应该觉得这是顺水推舟,而不是被说服。
