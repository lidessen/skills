# Skeleton · 重构后骨架

## Core claim

你每天用的 AI 工具正触及它的天花板,天花板不在模型能力、在"上下文"这一层。对话作为上下文主体本质是一个 **god file**——多条本该独立的脉络被挤进同一条扁平序列。下一步的 agent 就是把上下文当代码来组织:纵向分层、横向分模块。换了形态之后,机器什么都没变,LLM 仍然在做它唯一会做的那件事——续写。

## Audience

每天用 Cursor / Claude Code / Copilot 的前端工程师。懂软件架构、有过 god file 经验、对模块化有直觉,但没深入研究过 agent 内部机制。他们需要的不是 how-to,是 "aha——原来我熟悉的那套架构思维,正好用来理解这件事"。

## Reasoning chain

1. LLM 是续写引擎,让它做事 = 设计上下文(底盘)
2. 对话只是上下文的第一种形态,不是必然形态
3. 这种形态撑不住长任务——会膨胀、会稀释、会丢状态
4. 主流 harness 都在打补丁,但补丁都打在"对话是主体"这个前提上
5. 真正的出路是换前提——别让对话当主体
6. 换成事件流之后,LLM 那头没变,仍然是续写;变的是喂进去的文本长什么样
7. 有了新形态,就可以给上下文加架构:分层(信息密度)+ 分模块(职责边界)
8. 机制是通用的、200–500 行就能跑;真正的难点在为业务设计 schema

### §6 续写的不一定是对话 · ~200 字
- **Needs**: U-001 / U-002 回扣
- **Gap**: 需要原创一段。骨架:换掉对话、换上事件流,LLM 那头仍然是读文本预测下一段——变的只是那段文本长什么样。对话以"轮次"组织,每轮夹着礼貌措辞、过程性回复、工具原始输出;事件流以"状态变化"组织,每条都是裁剪过的高密度信息。对 LLM 是同一回事,对工程师是两种世界。
- **Function**: 防止读者把事件流当"新范式/新魔法";把续写视角贯通全文

### §7 上下文的架构化 · ~500 字

- **Function**: 给前端工程师最强共鸣的一节;把 god file 的解法翻译过来
