# Agent Harness 与上下文层级化

## 第一章 对话式 Agent:从预测机器到 Claude Code

### 1.1 一切的起点:LLM 是一台预测机器

LLM 本质上只做一件事:看一段文本,预测接下来应该是什么。

给它一段话,它输出下一个字;把这个字接在后面,它再输出下一个字。如此往复。模型内部没有"知识库"、没有"记忆"、没有"意图",所谓的回答、推理、调用工具,都是这台预测机器根据眼前的文本一字一字续写出来的。

这决定了所有上层系统的形态:**要让 LLM 做任何事,唯一的办法是把那件事的信息变成文本,放到它眼前**。

### 1.2 为什么 Agent 长成对话的样子

从一台预测机器到一个能完成任务的 Agent,中间要跨过一步:**任务是持续的,预测是一次性的**。一个需求往往要经过多个回合、多次工具调用才能落地,每一步都在产生新信息——补了一句话、某个工具返回了结果、模型做出了一个判断。这些新信息必须能在下一次预测时被看到,否则预测没法接着往下走。

怎么让新信息持续被模型看到?答案其实没什么选择:**把新信息接在已有文本后面,整段喂给模型**。

形式一旦确定,Agent 就长成了对话的样子:

- 用户说的话,是一条消息
- 模型的回复,是一条消息
- 工具调用和工具返回,也是消息

每一轮都在往这个消息序列后面追加内容。模型每次被调用,看到的是整个序列,然后预测下一条消息应该是什么。所谓"多轮对话",就是这个不断追加、不断预测的过程。

**对话不是界面设计选择,是 LLM 能承载任务的底层形态**。不是因为人习惯聊天所以 Agent 做成对话,而是因为预测机器只能这么用。

### 1.3 如何赋予 Agent 行动力:工具接入对话流

光会续写文本还不够,Agent 得能读文件、改代码、执行命令、查网页。这些动作要塞进一个只会预测文本的系统里,做法很直接:规定一种特殊的消息格式,比如"调用工具 X,参数是 Y"。模型输出这样一段文本时,harness(模型外面的运行程序)认出来,执行工具,把结果作为新消息追加回对话,再把整个序列交给模型。模型看到结果,继续预测下一步。

工具调用由此融进对话流。对话里不只有用户和模型的对话,还有模型和工具的对话。每一次工具执行,都是在对话序列里插一段新内容。

Agent 的完整循环于是成立:

1. 用户发出请求
2. harness 把当前对话序列喂给模型
3. 模型输出文本,或输出一个工具调用
4. 如果是工具调用,执行工具,把结果追加回对话
5. 回到第 2 步,直到模型不再调用工具
6. 把最终回复返回给用户

循环本身很简单,几十行代码就能实现一个能跑的版本。但能不能在真实世界里好好工作,是另一回事。

### 1.4 Claude Code:把对话模式推到极致

让这个循环真干活——改一个中等规模项目的代码——问题迅速暴露。

项目里几十个文件要让模型知道,一批工具可以用,这些信息全塞进对话,光是开场就占掉大半个上下文窗口。干到一半,某个工具返回几万行搜索结果,塞回对话后续轮就没空间了。搜索大量文件定位 bug 时,那些文件内容就能把上下文撑爆。对话跑得久了,早期内容越堆越多,模型开始抓不住重点。

**Claude Code 是这些问题的一整套工程回答**。每一项设计,都是在让对话流保持可持续。

**Skill:描述常驻,正文按需加载**。Skill 是一段针对特定任务的详细指令。一个项目可能注册几十个 skill,全塞进上下文,开场就爆了。Claude Code 只把简短描述放进对话——名字、干什么用的、什么时候用——只有模型判断这次需要某个 skill,它的完整正文才会被加载进来。

**Tool 定义:列表常驻,结果裁剪**。工具列表得让模型知道,所以一直在对话里。但工具执行结果不一定原样放回——搜索返回几万行时,harness 会截断或压缩。

**Subagent:独立上下文干脏活**。有些任务过程中产生大量中间信息,比如遍历大量文件。Claude Code 派一个 subagent 去干,subagent 有自己独立的对话流,干完只把结论告诉主会话,中间过程不进主对话。

**Compaction:满了就压缩**。对话跑久了,harness 自动把早期内容总结成摘要,保留关键信息,丢掉冗余细节,给后续腾出空间。

把这四件事放在一起看,**Claude Code 的工程重点不在模型,而在对话流本身**。它反复在做同一件事:每次调用模型之前,判断什么该在对话里、什么不该在、什么该压缩、什么该外包。

**harness 的核心是一台上下文组装机**。Claude Code 是这套思路目前做得最彻底的样本之一。

思路也不是 Claude Code 独有。OpenAI 的 Codex 现在官方定义为 harness,把"管理上下文窗口"列为核心职责。Nous Research 的 Hermes Agent、OpenAI 的 Agents SDK,做的都是同一件事。**Agent 不是一段 prompt,而是一套围绕对话流做信息管理的运行时系统**——这已经是行业共识。

### 1.5 对话跑久了会变成什么样

Claude Code 这样的 harness 把对话模式下能做的事情做到了很高的完成度。但"高完成度"只局限在一次对话的范围内——当对话本身跑得足够久,一个根本性的问题会浮现出来。

看一个实际的 context prompt 长什么样——下面是简化形态,结构贴近真实。

一次会话刚开始时,发给模型的上下文大致是这样:

```
[system]
You are Claude Code, an AI coding assistant...
<project_context from CLAUDE.md>
This is a Node.js backend project using Express and PostgreSQL.
Follow the existing code style in src/. Run tests with npm test.
</project_context>
<tools>
... (工具列表)
</tools>

[user]
帮我看看 user 注册接口为什么偶尔返回 500

[assistant]
我先看一下注册接口的代码。
<tool_use: read_file path="src/routes/auth.js">

[tool_result]
(auth.js 的 120 行代码)

[assistant]
看到了,问题可能出在数据库连接池配置上。让我再看一下 db.js。
<tool_use: read_file path="src/db.js">
...
```

干净、紧凑,每条消息都直接服务于当前问题。

对话进行 30 轮之后,同样的上下文结构会变成这样:

```
[system]
(不变)

[user] 帮我看看 user 注册接口为什么偶尔返回 500
[assistant] 我先看一下注册接口的代码。<tool_use: read_file...>
[tool_result] (120 行代码)
[assistant] 看到了,问题可能出在数据库连接池配置上。让我再看一下 db.js。
[tool_result] (80 行代码)
[assistant] 我发现连接池的 max 设置是 10,但是...(300 字解释)
[user] 那先把 max 调到 30 试试
[assistant] 好的,我来修改配置文件。<tool_use: edit_file...>
[tool_result] (修改 diff)
[assistant] 改好了,你测试一下。
[user] 好像好一点,但我想顺便让你看看 order 模块的查询也很慢
[assistant] 好的,我看一下 order 相关的代码。<tool_use: read_file...>
[tool_result] (200 行代码)
... (再过 20 轮)
[user] 对了前面 user 注册那个连接池的事,改成 30 之后还是偶尔报错
[assistant] 让我重新翻一下之前的上下文...
```

问题一目了然。

**真正有效的"状态信息"其实很少**。在这 30 轮对话里,对后续推进有价值的核心事实就那么几条:user 注册接口有间歇性 500 错误、数据库连接池 max 从 10 改到了 30、order 模块查询慢是另一个相关问题、前一个修改没完全解决问题。加起来不到 10 行。

**对话里实际占了多少 token?** 读入的文件内容、工具调用的结构化消息、模型的解释性回复、礼貌措辞、过程性的"好的我来看看""我发现""让我再检查一下"——全都在上下文里。几千 token 换几行有效信息。

**问题的根源是对话里每条消息同时在做好几件事**。一条模型回复同时承担四个角色——给用户看的表达、工具调用的发起、下一轮预测的输入、任务到目前为止的状态记录。四个角色对文本的要求不一致,写到同一段里,信息就被稀释了。

**这个问题在末端的具体任务执行上不明显**——任务短、回合少,稀释一点也够用。任务一拉长,问题就放大:对话流快速膨胀,膨胀的大部分不是有效信息,上下文窗口被低密度内容挤满,模型在里面找关键信息越来越吃力。

追问"Agent 能不能跨三天、跨多个话题地持续运行"——到这一步,对话模式已经走到了自己的边界。

## 第二章 把上下文从对话里抽出来

### 2.1 一个具体问题:让 Agent 跑三天

设想一个具体场景。

让 Agent 维护一个中型项目,为期三天。第一天上午派它排查用户注册的 bug,中午修完,下午派它加一个订单导出功能。第二天发现前一天那个 bug 其实没彻底修干净,回到那个话题继续;同时提一个重构请求。第三天要它把三天所有改动整理成一份 PR 说明。

这个场景并不极端,就是真实工作里普通的三天。但在当前的对话模式下,它非常难实现。

放在同一个对话里:上下文在第一天下午就膨胀严重,到第三天基本不可用。模型会丢失第一天早上的细节,或者早就被 compaction 压成了摘要。第二天提"前一天那个 bug"的时候,模型得花好几轮重新读代码才想起来是什么问题。

开新对话:模型每次都是一张白纸。它不知道前一天修过什么、改了哪些文件、为什么改。每次都得从头讲一遍——可自己都不一定记得全。

不是假想问题。任何想把 Agent 用在真实长期工作上的人,都会撞上它。

### 2.2 现有 harness 是怎么应对的

主流 harness 都意识到了这个问题,各自给出了答案。把这些答案放在一起看,差异小得可疑——它们都共享一个前提:**对话是上下文主体,要做的是让对话能跑得更久**。

**Compaction(对话内压缩)**。Claude Code、Codex 都有类似机制。上下文接近窗口上限时,harness 自动把早期内容总结成摘要,保留关键信息,丢掉冗余细节。这给当前对话腾出了空间,但压缩是有损的——被压掉的细节就回不来了,而且压缩出来的摘要本身也占空间。

**持久化 transcript(跨对话历史搜索)**。claude.ai 在产品层面提供过去对话的搜索,Hermes 类系统也有类似设计。新对话可以引用老对话里的内容。但这是"按需回查",不是"持续在场"——模型必须主动想起去搜,搜到的结果还是对话片段,密度不高。

**Memory 文件(显式知识外化)**。Claude Code 有两套:手写的 CLAUDE.md 和自动积累的 MEMORY.md。CLAUDE.md 是用户提前写好的项目指令,每次会话开始加载。Auto memory 是 Claude 自己在工作中识别出值得保留的事实(构建命令、调试经验、代码风格等),写进 `~/.claude/projects/<project>/memory/`,下次会话开始时加载前 200 行。更详细的内容拆到 topic 文件里,按需读取。Anthropic 的 API 还提供了一个专门的 memory tool,让模型主动管理 `/memories` 目录。

**跨对话 memory synthesis**。claude.ai 产品层面,每 24 小时会把历史对话合成一份关键洞察,作为新对话的背景。这是对"对话之间没有积累"问题的一个直接补救。

这些方案都在用,也确实有效。但把它们放在一起看,共同的模式很清楚:**都是对"对话这条线"的补充——在对话之外存点东西,或者把对话的起点变得更丰富**。核心工作流没变:主要交互还在一个对话流里进行,Agent 的主要上下文还是这个对话流。memory 是对话的起手素材,compaction 是对话的内部压缩,历史搜索是对话之外的回查通道。

所有方案都没有动一个底层设定:**上下文 = 对话**。

第一章已经点出,对话作为上下文主体有一个根本问题——信息密度低、粒度太细,因为每条消息同时承担多重职责。memory 和 compaction 能让对话跑得更久,但不能让对话变得更"浓"。膨胀被推迟,没有被消除。

补丁打了这么多,对话还是那个对话。

要真正解决跨三天、跨话题的问题,得换一个思路:**不要让对话当上下文主体**。

### 2.3 换一种结构

重新看 context prompt。

原本的结构是这样(简化):

```
[system]
(系统指令 + 项目说明 + 工具列表)

[user] ...
[assistant] ...
[tool_use] ...
[tool_result] ...
[user] ...
[assistant] ...
...
```

对话序列是上下文的主体。每一轮 user/assistant/tool 的往返,都在给这个序列追加内容。Agent 的"长期记忆"——它知道这个项目正在进行什么、改了什么、遇到过什么——全都绑在这个序列里。

换一种结构:

```
[system]
You are the orchestrator for a long-running project...

[system notification]
新任务:用户报告 user 注册接口间歇性 500 错误。
派发 worker-01 处理。

[system notification]
worker-01 报告完成:
- 问题定位:数据库连接池 max=10 过低
- 修复:src/config/db.js 中 max 调整为 30
- 验证:通过本地压测,错误率从 3% 降至 0
- 风险:生产环境连接数限制需确认

[system notification]
用户反馈:前述修复未完全生效,仍有偶发错误。
派发 worker-02 深入排查。

[system notification]
worker-02 报告完成:
- 根因:除连接池外,还有一处 transaction 未正确释放
- 修复:src/routes/auth.js 第 87 行补充 client.release()
- 验证:连续 10000 次请求无错误

[system notification]
新任务:用户请求添加订单导出功能。
派发 worker-03 处理。
...
```

这个上下文发生了根本变化:**对话不见了**。取而代之的是一个事件流——系统通知、任务派发、完成报告。每一条都是高密度信息,没有礼貌措辞、没有过程性表达、没有工具调用的原始输出。

对话去哪了?被推到了下一层。每一个"派发 worker-XX 处理"对应的,是一个独立的对话式 Agent 去完整执行那个任务。worker 内部用的还是传统的对话上下文——工具调用、模型回复、往返多轮,和 Claude Code 现在做的事一样。worker 干完之后,它的几千 token 对话流被压成一条几十字的"完成报告",作为一条事件写入上一层的事件流。

两层上下文做着不同的事:

- **下一层(对话式 worker)**:负责具体任务的执行。高频、细粒度、对话形态。每个 worker 的生命周期就是一个任务——任务结束,上下文销毁,只留下一份报告。
- **上一层(事件流)**:负责长期的统筹。低频、高密度、事件形态。每一条事件代表"已经发生的、对后续有影响的"一个节点。

上一层的上下文积累速度慢得多。上一层的 1M token 能容纳几百个 worker 任务的完成报告。下一层每个 worker 的 1M token 只服务于当前那一个任务,任务结束就清空。

### 2.4 上下文的层级化:不止两级

上面的例子是两级。真实工程应用里,两级往往不够。

从最底层开始编号:

- **L0**:最具体的一线执行层。对话式 Agent,直接调用工具、读写文件、完成一个具体任务。上下文形态是传统的 user/assistant/tool 对话。生命周期短,任务结束即销毁。
- **L1**:L0 的统筹层。上下文是事件流,每一条事件是一个 L0 任务的完成报告。L1 决定派发什么任务、拆解什么目标、什么时候把阶段性结论汇报给更上层。
- **L2**:L1 的更上层。项目足够复杂时,L1 本身也会积累大量事件流,这时候 L2 可以把 L1 的事件流进一步抽象——把"过去一周 L1 处理的所有 bug 修复事件"合成一条"Q3 bug 修复阶段完成"的更粗事件。
- **更多层**:看场景需要。小项目两级够用,复杂项目可能需要更多层。

分层的逻辑类似公司的管理结构。一线工程师每天处理具体 ticket,感知粒度是"这个函数的这个 bug";部门经理感知粒度是"这个模块这周的进展";总监感知粒度是"这个产品线这个季度的方向"。不同层感知的信息粒度不同、节奏不同、覆盖的时间跨度不同。不是说总监"比工程师知道得更多",而是**总监不应该知道每个 ticket 的细节,他该知道的是被压缩过的高层事件**。

上下文的层级化是同样的道理。每一层的上下文只装它这一层该看的东西。L0 装对话细节,L1 装事件报告,L2 装阶段总结。往上每升一层,信息密度更高、积累速度更慢、时间跨度更长。

没有通用的层级划分。有的场景三级够,有的场景需要五级。怎么分、分几层、每一层装什么粒度的信息——这是第三章和第四章的问题。

### 2.5 为什么这样能解决跨三天的问题

回到 2.1 的场景:三天、跨话题、需要持续积累。

在分层结构下,三天发生的事情大致是这样:

- **L0** 层有过几十个短生命周期的对话——每个 bug 排查一个、每个功能实现一个、每个重构一个。每个 L0 在自己的任务内部用对话上下文完成工作,完成后销毁,只留一份报告。
- **L1** 层有一个长生命周期的事件流,积累了这三天所有的任务报告。到第三天派它"整理 PR 说明"时,L1 不需要回忆——它的上下文里就是三天的事件流,每一条都是高密度的完成报告。
- 第二天提"前一天那个 bug 没修干净"时,L1 直接从事件流里找到前一天的 worker-01 报告,派一个新的 worker-02 带着这份报告去继续排查。worker-02 是新开的对话,但它知道前因后果——因为 L1 给了它。

两个关键变化:

**上下文不再膨胀**。L1 的事件流增长非常慢——三天几十条事件,每条几十到几百字。对比之下,三天的原始对话流可能是几十万字。

**对话质量不再退化**。L0 的每个对话都是短的、聚焦的、新鲜的。没有哪个对话需要背负三天的历史。对话回到了它最擅长的场景:末端的具体任务执行。

这个思路的核心不是"发明了什么新东西",而是**把本来被混在一起的两件事分开**:对话擅长任务执行,事件流擅长长期积累。现有 harness 让对话同时干这两件事,所以两边都不够好;分到不同层,各干各的,两边都回到了舒服的工作区间。

由此引出一系列实现问题:

- 事件流里的"事件"到底长什么样?什么是一个好的事件粒度?
- L0 worker 怎么被派发?怎么接收 L1 传给它的上下文?
- L0 完成后,它的对话流怎么被压成 L1 的一条事件?谁来压,怎么压?
- L1 自己什么时候决定派新 worker,什么时候直接回答,什么时候向更上层汇报?
- 多层之间的信息流怎么设计,才能既不丢失关键信息,又不把低层细节泄漏到高层?

这些是第三章要回答的问题。

## 第三章 从对话流到工作区信息流

### 3.1 上层的"文本"从哪儿来

第二章画了一张分层图,但一个基本问题还没答:L1 那条事件流,到底是什么?

回到第一章的起点。LLM 是预测机器,它永远基于一段文本做预测。上一层的 Agent 也是一个 LLM,换不了这个事实——它运行的唯一方式,还是把一段文本喂进去,让它预测下一步该干什么。

分层并不是"高层 Agent 不看对话了,改看别的"。分层的实质,是**把喂给高层的那段文本换一种组织方式**。

换成什么?一条**工作区信息流**——把项目当前正在发生什么,以客观记录的形式排成一串,交给高层 Agent 阅读。它在形式上和对话一样都是文本,但内容不再是对话往来,而是事件。事件里有什么?任务被派出去了、worker 回来了、外部推了一个告警、用户提了一个新要求。高层 Agent 读这段文本,基于它判断下一步。

这个改动不是发明。对话流里本来就有这些信息——提了什么、任务派给谁、结果是什么,都在里面。问题不是"缺信息",而是信息被礼貌措辞、过程性回复、工具原始输出稀释到无法直接使用。工作区信息流做的事是**把那些被稀释的核心信息单独抽出来,以固定形式排好**。

第二章结尾留了五个开放问题。接下来按顺序回答。

### 3.2 事件长什么样

一条事件,是一次**状态变化的最小可陈述单元**。具体一点:

- "用户反馈 user 注册接口仍有偶发错误" 是一条事件
- "派发 worker-02 排查 transaction 释放问题" 是一条事件
- "worker-02 完成,定位到 auth.js 第 87 行漏 release,已补齐" 是一条事件
- "监控告警:订单系统 5 分钟内错误率 0.3% → 1.2%" 是一条事件

不是一条事件的例子:

- "让我看一下代码"——这是过程,不是状态变化
- "好的,我来处理"——这是礼貌,不是状态变化
- "<tool_use: read_file path=...>"——这是动作,不是它的结果

事件粒度的判断标准只有一条:**离开上下文后这条事件是否还能独立被理解**。"worker-02 完成任务" 不能独立理解——要结合前一条才知道任务是什么。合并成 "worker-02 完成 transaction 释放修复" 就能独立理解了。

事件不限于 agent 内部。外部世界也产生事件:webhook、CI 告警、监控通知、用户消息。它们以同样的格式写进事件流。用户在聊天里说的话,对事件流来说也只是 "用户在 t1 提出了 X" 这样一条事件——具体措辞被压掉了,意图被留下。

事件流不是对话流去了礼貌。它是**另一种文本**,结构和信息密度都和对话根本不同。

### 3.3 Worker 怎么派发

L1 决定派一个 worker 去干活时,要回答两个问题:给它什么目标、给它什么背景。

**目标**是一句话的任务描述。"排查 user 注册接口 transaction 释放问题"。不要写得含糊——worker 拿到模糊目标只会花迭代预算去猜。

**背景**是从 L1 的事件流里挑出的相关事件。怎么挑?只挑与当前任务直接相关的:worker-01 的报告、用户原始反馈、涉及到的代码位置。不相关的全部排除,哪怕它们在事件流里离当前任务很近。

这一步是层间信息流的关键。**L1 给 worker 的不是事件流本身,是事件流的投影**——一个为这个任务裁出来的子集。

Worker 拿到的是什么?一个 fresh 的对话上下文。第一条 system/user 消息里装着目标和背景事件。它不知道 L1 的完整事件流存在,也不需要知道。在它自己的世界里,它就是一个普通的对话式 agent:读文件、改代码、跑命令、回结果。Claude Code 的 subagent 已经是这个形态,分层架构把它推到极致——**每一个 L0 都是临时 worker,任务结束就消失**。

Worker 不会记得前一个同类任务发生过什么。记忆在 L1,不在 worker。

### 3.4 把对话压成事件

Worker 干完活,它自己的对话上下文可能有几万 token——读了十几个文件、跑了几轮工具调用、出错重试过两次。这些东西不能原样进 L1。

压缩过程是这样。

Worker 结束时,交出它的完整对话流。调用一个独立的模型(不是 worker 本身),按**固定 schema** 抽取几项内容:做了什么、结果是什么、改动了哪些文件、遗留了什么风险、未完成的事。几十到几百 token 的一条结构化报告。

**压缩不是摘要,是抽取**。摘要是把长文章变短,内容还是连续叙述;抽取是按预设字段填空,输出格式是固定的。对后续 L1 使用来说,结构化输出比自由摘要好用得多——要查"哪些文件被改了",直接看字段;要判断"这个任务是不是真完成了",看"遗留风险"和"未完成的事"两项。

压缩为什么要独立调用?因为 worker 本身带主观色彩。跑到最后它可能已经在解释性地回答用户,让它"总结一下自己做了什么",它会倾向性地美化——就像工程师写周报。独立调用拿着冷冰冰的对话流抽字段,不带解释。

压缩出来的这条结构化报告,就是写进 L1 事件流的那条事件。

### 3.5 L1 每轮做什么

L1 自己也是一个 agent,也有一个循环。每轮它看到自己的事件流,预测下一步。下一步有三种形态:

- **直接响应**。事件流里已经有足够信息回答当前问题,就不需要派 worker。用户问"昨天那个 bug 修了吗?"——事件流里就有 worker-02 的完成报告,L1 直接答。
- **派新 worker**。需要外部动作(改代码、读文件、跑命令、调外部 API),就裁好上下文派下去。
- **向上汇报**。事件流积累到某个阶段性节点(一个 milestone 完成、一个 sprint 结束、一个 bug 排查告一段落),或者事件流本身开始膨胀需要压缩,L1 向 L2 推一条抽象更高的事件。

三种动作不是 if-else 代码写死的逻辑,是 L1 基于事件流"预测"出来的。给 L1 的 system prompt 里说明这三种动作的边界和可用工具,剩下它自己判断。L1 是一个 agent,不是一个规则引擎。

一个容易走歪的方向是给 L1 塞大量决策规则。"事件超过 50 条就要压缩"、"worker 执行超过 10 分钟要汇报"。规则加多了,L1 就不再是 agent,是一个用 LLM 当 CPU 的复杂 workflow 引擎。agent 的价值在于它能处理规则外的情况——与其列举,不如让它看事件流自己判断,出问题再调 prompt。

### 3.6 防止泄漏和丢失

多层系统有两个对称的风险:**细节向上泄漏**,**信息向下丢失**。

**泄漏**的典型症状是 worker 的对话内容、工具原始输出、模型解释性回复跑到了 L1 的事件流里。L1 上下文就不纯了,又回到对话膨胀的老路。

防泄漏靠两道闸。第一道,worker 完成后的压缩调用是强硬边界,只有压缩出来的结构化报告进 L1,中间产物一概不进。第二道,L1 自己不读 worker 的原始对话。如果 L1 发现报告信息不够,它的动作是**再派一个 worker 带着更具体的目标去重做**,不是把上一个 worker 的日志调出来翻。

**丢失**的典型症状是 worker 发现了一个重要风险("这个修复依赖某个环境变量,生产可能失效"),结果压缩把它压没了,几小时后问题爆发。

防丢失靠压缩 schema 本身。schema 强制包含"遗留风险"和"未完成的事"两个字段,**即使 worker 主观认为修复 OK,只要它在对话里提过风险,压缩调用就必须填进去**。worker 不能声称"没有风险"然后结束——压缩调用看不到这个声明,只看事实陈述。

层与层之间还有一个方向是横向——同一层多个 agent 之间。这里暂时不展开。要协作的话就走事件流:两个 worker 不直接对话,通过读 L1 事件流得知彼此进展。这是下一步的设计,不是本章的重点。

### 3.7 最小可运行形态

上面这套机制听起来不少,落到代码其实很小。一个最小跑起来的分层 harness 需要的东西:

- **一个 L0 runtime**。复用现成的就行——Claude Code、Codex、Agents SDK,任何一个能跑对话式 agent 的 runtime 都可以。
- **一个事件流存储**。一个 append-only 的 JSONL 文件或 SQLite 表,每条记录是一个结构化事件。
- **一个 L1 循环**。读事件流、调模型、做三种动作之一(响应/派发/汇报)。几百行代码。
- **一个压缩调用**。Worker 结束后,用固定 prompt 把对话流抽成事件,写回存储。

整套原型 200-500 行代码能跑。跑起来之后真正复杂的地方不在机制,在**事件 schema 的设计**——什么该被当事件、什么字段必填、怎么命名。这才是工程量最大的一环,因为它需要对具体业务领域的理解。

这就自然过渡到了第四章。分层 harness 不是一个独立的产品,它必须嵌在一个具体的业务系统里才跑得起来。

## 第四章 落到业务:数字流水线

### 4.1 分层 harness 需要什么样的业务环境

分层 harness 再精巧,单独放出来只是一个空壳。它要发挥作用,得有一个"高度信息化"的业务环境。

"高度信息化"不是一个空洞词,具体指两件事。

第一,**业务状态对 agent 可读**。客服系统、工单平台、代码仓库、监控面板、项目看板——每一个业务节点的当前状态,agent 能通过某个接口读取。MCP 协议正在做这件事:把每一类业务系统的信息暴露成标准的 tool 和 resource。

第二,**业务动作对 agent 可写**。回工单、回复客户、提交代码、改配置、触发流水线——每一个可能的操作,agent 能通过 API 调用完成。

两个条件缺一不可。只读不写,agent 是一个分析器,生成报告给人看;只写不读,agent 是一个傀儡,完全依赖外部告诉它当前状态。

很多公司想把 agent 接进来但效果不好,根源往往不在 agent 自己,而在业务系统——状态不结构化、动作没有 API、每次调用都要 session 认证。基础设施不齐,分层 harness 再好也落不下来。

### 4.2 工位化:agent 应该接管什么

一个常见错误是想让 agent "做某个岗位的全部"。"AI 客服"、"AI 工程师"、"AI 运维"——听起来很酷,实际做起来 scope 太大,效果飘忽。

更实际的做法是**工位化**。把岗位拆成一个个工位,每个工位有三件东西:

- 明确的输入事件(工单创建、代码提交、告警触发)
- 明确的输出动作(回工单、改代码、升级告警)
- 明确的完成标准(工单关闭、PR 合并、告警消除)

这种工位,agent 能覆盖。工位越清晰,agent 做得越稳。一个普通公司里这样的工位其实非常多——填一份表、处理一条 L1 工单、跑一次周期性报表、在代码库里做一次版本号升级。每一个单独拿出来都不够成一个岗位,但加起来占了大量人力。

工位化还有一个副作用:它**自然形成事件 schema**。一个工位的输入是一类事件,输出是另一类事件。把工位定义清楚,事件流的结构就定下来了。事件 schema 不是从架构角度先设计出来的,是从业务工位自下而上长出来的。

### 4.3 人做什么

工位被 agent 接管后,人不是消失,是上移一层。

人做三件事:

- **制定策略**。哪个工位开放给 agent、开放到什么程度、什么情况需要回退到人。
- **审核关键节点**。一个工位的完成标准里,某些 action 需要人确认才能生效——改生产配置、发对外声明、删数据。
- **处理异常**。事件流出现 agent 处理不了的模式,触发人工接管。

这不是"人监督 agent 每一步",而是"人定义什么叫对、什么叫边界,agent 在边界内执行"。权限分层在 Claude Code 里已经有雏形——default / acceptEdits / auto / bypassPermissions 这几档,本质是不同敏感度的动作走不同的审批路径。这套思路推广到业务,就是每个工位定义自己的敏感度档位。

### 4.4 数字流水线

多个工位 agent 通过事件流协作,一条业务流程可以被完整覆盖。

以一个客户投诉处理为例:

1. 客服工位 agent 收到客户消息(事件:用户输入),提取意图,创建一条工单(事件:工单创建)
2. 工单分类 agent 订阅"工单创建"事件,判断类型,路由到对应的处理工位(事件:工单分配)
3. 处理工位 agent(可能是技术支持,可能是订单修改)接到分配,执行动作(事件:工单处理中 / 工单已解决)
4. 客服工位 agent 订阅"工单已解决"事件,给客户回一条完成消息
5. 复盘工位 agent 每天跑一次,扫最近所有工单的事件,生成统计报告

五个 agent 之间**没有一句对话**。完全通过事件流协作。每个 agent 只订阅自己关心的事件类型,产生自己输出的事件。中央没有调度器,没有主 agent——事件流本身就是调度。

这种结构像物理流水线:每个工位只负责自己那一步,上一步交下来的东西有固定接口,下一步拿走的东西也有固定接口。中间没有会议,没有交接邮件。差别在于这条流水线跑的是数字事件,全过程可追溯。

### 4.5 小团队的机会在哪儿

基础设施层——大模型、通用 harness、通用记忆系统——大厂会持续投入。Anthropic、OpenAI、Nous 都在做。再过一两年,分层 harness 本身也可能变成开源基础组件。

这些基础能力变成商品之后,真正的价值在**业务层**:

- 某个行业的工位应该怎么拆
- 某个业务的事件 schema 应该是什么
- 某个工作流的敏感度档位应该怎么定
- 哪些工位适合 agent,哪些不适合

这些问题,大厂的通用方案答不上来。需要对具体行业的深度理解——客服行业和 devops 行业需要的工位完全不同,同样是客服行业,SaaS 和电商的事件 schema 也不一样。

这就是小团队的位置:**不做通用 agent,做"这个行业的数字流水线"**。通用层交给大厂,业务层靠领域内的人自己长出来。分层 harness 只是一个承载容器,真正决定效果的是容器里装的是什么。

### 4.6 结论

回到全书开头那个基本观察。LLM 是预测机器,它基于一段文本预测下一段。上层系统的所有设计,都是在回答"这段文本应该是什么"这个问题。

对话是这段文本的第一种形态。Harness 围绕对话做了一整套治理——懒加载、压缩、隔离、子代理。这些机制让对话撑得更久,但没有改变对话是上下文主体这个底层设定。

分层的意思,是让这段文本从对话变成事件流。对话继续存在——被推到最底层 L0,服务于具体任务的执行。但长期积累的部分,用事件流做,信息密度高几个数量级。

事件流不是一项新发明。它只是把本来被稀释在对话里的信息抽出来排好。真正的门槛不在这套机制,在**为具体业务设计对应的事件 schema**。机制是通用的,schema 是领域的。

这个方向会把 agent 的角色重新定位——从陪聊助理变成长期驻守在工位上的执行者,和人组成一条数字流水线。流水线已经存在,它就是 Claude Code、Codex、Hermes 里一块一块长出来的那些机制。接下来不是再发明什么,是把这些零件拼起来,放到具体业务里,看它能不能真正跑起来。
