# project-restructure:verify — Check the wave against hard constraints

Verify the most recently executed wave from `restructure/PLAN.md`. Produce
`restructure/VERIFY.md` or append a dated section to it.

## Checks

Run only the checks named in the plan, plus these baseline guards:

1. **Semantic source** — `principles/SEQUENCE.md`, adopted design decisions,
   and declared canonical trees still exist and were not replaced by projections.
2. **Inheritance** — adopted records, reviews, and any named preserved seeds are
   still reachable with source links intact.
3. **Projection truth** — if projections were synced, they match canonical
   sources for the rows in scope.
4. **Snapshot consistency** — for each sequence-dependent skill touched, packaged
   snapshots were regenerated by `scripts/sync-sequence-snapshot.py` or
   intentionally left unchanged with reason recorded.
5. **Index surfaces** — `README.md`, skill tables, and command lists describe the
   post-wave reality.
6. **No shadow authority** — no research note, blueprint, or stale doc now acts
   as if it were the sequence or design source.

## Verification artifact

```markdown
# Restructure Verification

**Wave verified:** <name>
**Date:** <ISO date>
**Plan basis:** restructure/PLAN.md

| Check | Result | Evidence |
|---|---|---|
| ... | pass / fail / not run | <path, command, or observation> |

## Verdict

pass | blocked | fail

## Next action

<continue to next wave | rerun inventory | route to other skill | human decision>
```

A wave is `pass` only when every required check for that wave passes. A failed
inheritance or semantic-source check is always `fail`, not “mostly done”.

## Stop conditions

- Do not start the next wave on a `fail` or `blocked` verdict.
- Do not claim project-wide cleanup complete until all planned waves are verified
  or explicitly deferred in the plan.
