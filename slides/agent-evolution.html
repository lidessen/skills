看 Agent 的演化

LLM 只会做一件事:看一段文本,预测下一段。 所谓 Agent、工具调用、记忆、多轮对话, 在模型这一端都是同一回事——续写一段文本。 这些年所有看起来是"能力跃迁"的变化,其实都发生在这段文本上。

下面用同一个场景——"LoginForm 的 Submit 按钮偶尔点了没反应"——看同一份上下文,如何在四个时代演化。

Gen 01 纯对话 2022
system user assistant

第一代 · 2022两个人的对白

GPT 刚出,ChatGPT 把 LLM 包装成聊天界面。模型眼前的上下文里,只有 两类消息——user 和 assistant——交替追加。

每一轮,模型都看到全部历史,然后预测下一条。

事实,靠人手动粘进来

模型没办法自己读文件,不能执行命令,也看不到磁盘。你要它分析的每一段代码、每一段日志,都必须你先复制,再粘贴到对话框里。

本质 上下文里只有人的话和模型的话。事实的传递完全由人中转。

Gen 02 对话 + 工具 2023
system user assistant tool_use tool_result

第二代 · 2023机器开始插嘴

Function calling 出现。上下文里多了一种消息:模型可以发起动作,动作结果以消息形式追加回来。

事实不再由人粘进来——模型自己用工具取。

两条循环并存

同一条时间线上现在有两条循环:

外循环:user ↔ assistant,和人的对话。
内循环:assistant ↔ tool,agent 自己产生、自己消费的动作与结果。

本质 两条循环都往同一个扁平序列里追加。这个设计成了所有 agent harness 的默认形态。

Gen 03 工程化分区 2024 · 2025
system memory compacted user assistant tool

第三代 · 2024–25把"对话 + 工具"推到极致

Claude Code、Codex、Cursor Agent 把这套推到了极致。实际喂给模型的上下文已经远不止前两层——中间多了一整套工程化的分区。

几样东西被加进来

CLAUDE.md — 项目常驻指令。
Auto memory — 跨会话积累的偏好、事实。
Skill 描述常驻 / 正文懒加载。
Subagent — 脏活派给独立上下文,回传结论。
Compaction — 接近窗口上限时自动压摘要。

补丁都打在外循环上

对话跑到 30 轮,系统会把前 20 轮压成一段摘要,塞回上下文前部——细节消失了。你再提起之前的连接池问题,模型要么瞎猜,要么重新读一遍代码。

本质 所有这些机制共享同一个前提:对话是上下文主体。补丁都打在外循环上。

一条消息承担四件事

给用户看的表达、下一轮预测的输入、工具调用的发起、任务状态的记录——四件事对文本的要求不一致,挤在一段里互相稀释。

30 轮之后,几千 token 换几行有效事实。追问一句 "能不能跨三天、跨多个话题地持续运行"——这个形态就撑到头了。

Gen 04 上下文分层 exploring
sys notify dispatch report status compacted

第四代 · 探索中不要让对话当主体

换个思路:把对话推到最底层,只服务于"执行具体任务"。上一层换一种文本形态做长期积累——事件流。

像把一段很长的代码逻辑抽成函数:调用方看到的是签名,不是函数体。

L1 只记录一次 session 的"签名"

每一次 L0 session run 结束时,由独立模型把整段对话压成一条结构化事件写回 L1——topic、turns、tools、risk。原始 user 消息、来回、措辞,全留在 L0。

L1 看到的是函数签名,不是函数体。这是 L0 和 L1 之间那道折叠。

L0 就是一次普通的 session run

别把 L0 想成"sub-agent"——它就是你今天用的那种对话:user、assistant、工具调用、来回澄清。L1 的某一行展开,就是一次完整 L0。

sub-agent 是另一种形态的 L0:少交互、以执行任务为主。同一层,不同用法。

L2 及以上:按需引入

L2 不是默认存在的。只有当场景足够工程化——比如要做周报、模块历史、跨版本记忆——才会在 L1 之上再叠一层。密度更高、积累更慢、时间跨度更长。

本质 每一层都是一次可按需展开的折叠;叠多少层,由场景决定,不是必选项。

Agent 的下一步,不在模型变强,在 这段文本继续被组织得更好。

事件流不是一项发明,只是把原本稀释在对话里的信息抽出来排好。真正的门槛在为具体业务设计对应的事件 schema——机制是通用的,schema 是领域的。
