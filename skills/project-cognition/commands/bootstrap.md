# Bootstrap Project Cognition

Create a reusable working model only after the persistence gate passes.

1. Name two or three concrete future decisions the model must support, the
   later actors, the authoritative sources, and the fixed source revision. If
   these cannot be named, return a temporary orientation instead of files.
2. Inspect the project whole at low resolution: governing design, major
   responsibilities, public surfaces, state and effect owners, verification
   surfaces, and the main execution or information paths. Record absent or
   contradictory sources rather than filling them by convention.
3. Read `references/decomposition.md`. Propose the smallest set of coherent
   semantic questions whose answers cover the named decisions. Explain why this
   project produces these cuts. Include cross-packet relations and local
   acceptance before execution.
4. Execute directly or hand prepared packets to the caller's available carrier.
   Preserve source revision and packet identity. If one packet overloads or
   fails, retain successful siblings and repartition only that semantic region.
5. Synthesize by reconnecting responsibilities, causal/state paths, authority,
   contracts, and unresolved boundaries. Check decision-relevant claims against
   source; disagreement remains visible until evidence resolves it.
6. Read `references/projection.md`. Follow an existing target-project convention
   for derived knowledge. If none exists and persistence is authorized, use a
   plainly discoverable `docs/cognition/` directory with `INDEX.md` plus
   meaning-owned Markdown facets. Record the location in the project's normal
   agent-discovery surface when that surface already exists; do not create a
   new context system merely to advertise it.
7. Prepare the projection for an external check against at least one previously
   unstated downstream question. Retain what source the bootstrap agent checked,
   what changed the answer, and remaining uncertainty, but do not call that
   self-check independent verification. Name the actual external reviewer, run,
   or governed process only after it acts; otherwise record `unverified` and who
   may later verify, accept, or commit the files. Never put the Skill itself in a
   verifier field.

Do not claim a complete project model. Report the named decision coverage and
known blind spots.
