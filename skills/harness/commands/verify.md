# Verify a Harness

Verify that a harness reaches an actual agent and changes the intended action.

1. Read references/runtime-capability-map.md and references/evaluation.md.
   State the runtime, the declared loading path, the target agent action, and a
   result that would disconfirm the harness claim.
2. Name a durable evaluation or state artifact and a pointer from the harness
   before running the probe. Choose a real task that requires one load-bearing
   project judgment, such as respecting a documented boundary or applying a
   non-obvious convention. Give the agent only the normal entry surface; do not
   paste the expected answer.
3. Compare the same acceptance condition in a controlled baseline that lacks
   the target harness route and in the ordinary harness-enabled entry. If a
   safe baseline is impossible, state why; the result can show compatible
   behavior but not attribute improvement to the harness.
4. Run a nearby boundary task that should not need the harness. Confirm that it
   does not cause needless loading, misrouting, or broad project intervention.
5. Observe the actual context path where the runtime permits it. If the path is
   opaque, record what is proven by the task result and what remains unknown;
   do not convert an inference into a loading fact.
6. Record the baseline/action, boundary, and context outcomes in the durable
   artifact, with its discovery pointer, their evidence, and the
   smallest repair or next probe. Retest after a material layer or runtime-map
   change.

A passing configuration parser, hook registration, or prompt review is only a
structural check. It does not settle this verification.
