# Context Engineering Concepts

## In plain words

Context engineering connects an existing source of meaning to the agent action
that needs it. It asks what must arrive, when it must arrive, and which real
runtime surface can carry it without copying authority into a new place.

## Essence

1. **Target action** — the concrete judgment or operation that the information
   must change.
2. **Authoritative source** — the governed artifact that owns the meaning or
   fact; delivery does not rewrite it.
3. **Delivery projection** — an instruction, skill entry, pointer, retrieved
   fragment, or summary that makes source material available to an agent.
4. **Activation moment** — the point at which the information can affect action.
5. **Observed path** — task-specific evidence that the runtime actually carried
   the projection to the target agent.

## Delivery timing

Use three questions when they help, without requiring three named layers:

| Timing | Question | Common forms, when supported |
|---|---|---|
| Before scoped action | Must this orient every relevant action in this project? | root instructions, system context |
| On activation | Is it needed only after a task or method is selected? | skill prompt, task packet, routed policy |
| On demand | Is it useful only for one concrete detail or evidence check? | linked reference, retrieval, tool read |

One runtime may collapse these timings or expose more precise surfaces. The
timing relation is durable; L1/L2/L3 is only one possible projection of it.

## Boundaries

- Creating or accepting source content is domain work.
- Deciding source authority and lifetime is artifact organization.
- Engineering a reusable skill expression is skill engineering.
- Implementing hooks, memory stores, retrieval services, or agent loops is
  runtime or project-adapter work.
- Context engineering selects and verifies the delivery relation among them.
