# Context Engineering Evaluation

Retain only probes that can change a placement or deployment decision.

| Probe | Question | Disconfirming observation |
|---|---|---|
| Action | Did the ordinary agent entry use the delivered information in the named judgment? | The action still misses the boundary or requires pasted hidden context. |
| Boundary | Did a nearby task avoid irrelevant delivery or project-wide intervention? | The same context loads or triggers where it cannot change action. |
| Capability | Does the selected runtime actually support the proposed surface? | The claim comes only from another tool, version, or familiar filename. |
| Path | Did this task receive the named source or projection at the required moment? | Only general documentation or answer plausibility supports loading. |
| Authority | Does the projection remain linked and subordinate to its source? | Copied context silently becomes independent doctrine or accepted fact. |

Use the same target action and acceptance condition for a baseline comparison.
Without a comparable baseline, evidence may support delivery and compatible use
but cannot attribute improvement to the context change.

Persist a result only when handoff, approval, or later revision needs it:

```text
Target agent, runtime, and action:
Authoritative source:
Delivery timing and surface:
Capability and path evidence:
Action and boundary observations:
Authority check:
Verdict and attribution limit:
Smallest repair or reopening observation:
```
