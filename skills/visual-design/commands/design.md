# Design the Artifact or System

Use this path for the common create, redesign, or implementation task when the
deliverable is a real interface, document, illustration, or visual-system
slice. It may shape a missing case locally, but it does not run project-level
cultivation unless the caller asks to establish a continuing project direction.

Keep the frame thin and the artifact loop strong. Read enough to protect the
content, direction, and negative boundary, then make and inspect a real
fragment. An exhaustive design explanation before the first render is not a
substitute for designing.

1. Read the owning content/product source, existing direction or Aesthetic
   Case, the actual implementation owners, and one representative render when
   rendering is available. Start the baseline render while inspecting source
   rather than postponing observation until all optional references are read.
   Retrieve a provisional visual-language guide or design-system detail only
   when it can still change the next decision. Confirm that requested tools and
   write paths are available. Report a missing capability instead of
   simulating completion. When a logo, icon, illustration, repeated scene, or
   motion asset is a primary production object, read
   `references/visual-asset-production.md`; do not load it for ordinary
   interface work merely because the interface contains icons or imagery.
   If the request supplies only an unresolved style cue and design must proceed,
   read `references/visual-language.md` and form the smallest local candidate
   comparison needed before implementation. Do not silently choose the most
   familiar treatment.
2. When content hierarchy, attention, layout, or a token system is unresolved,
   read `references/presentation-model.md`. Inventory content, actions,
   relationships, variability, and states. Define the attention path:
   orientation, primary action, supporting judgment, context, and recovery.
   If the visual relation remains unresolved, form the smallest local contrast
   and negative boundary needed to proceed. When the host has no useful design
   guidance and the task requires a new hierarchy, layout, or token system,
   read `references/design-sources/index.md`, select one to three complementary
   sources, and read only their detail files. Record the decision each changes
   and what must not be copied.
3. Translate the direction and any provisional visual language into working
   decisions for this object. When the direction is accepted, compile the
   smallest **project visual contract** needed by this object. When it is still
   provisional, keep the same decisions as a case-scoped implementation
   hypothesis; do not persist or name them as project authority merely because
   implementation must proceed. Preserve the
   intended relation, expression allocation, shared invariants, open
   dimensions, and negative boundary. State what identity, orientation,
   primary action, ordinary content, functional controls, feedback, and
   recovery must express strongly, quietly, differently, or not at all. Retain
   exact values and component decisions only when an owning source or accepted
   representative fragment establishes them. Place durable additions in the
   project's existing design source; do not create a new contract artifact or
   schema by default.
4. Treat the existing implementation as evidence, not as an invariant. Preserve
   content, actions, functional contracts, and accepted relations; a redesign
   may replace the current composition, component anatomy, and treatment. Only
   when recurring component conformance is the principal unresolved judgment,
   read `references/component-expression.md`. Otherwise identify the real
   implementation owners and affected consumers in compact working notes and
   continue. Reject choices justified only by trend or the surface of an
   unrelated successful artifact.
5. Choose one decisive relation and make the smallest representative fragment
   carry it through the layers that matter: content grouping, layout,
   typography, component anatomy, material, imagery, or motion. This is an
   implementation wager, not a miniature style guide. For a broad redesign,
   do not assume palette replacement is sufficient; justify a palette-only
   result with rendered evidence that the existing structure and component
   expression already carry the direction.
6. Derive layout from content relationships and the attention path. Establish
   reading order, grouping, hierarchy, density, responsive behavior, and the
   smallest identity layer shared by in-scope surfaces. Let page archetypes and
   task states differ where their functions require it.
7. Reuse or create semantic tokens only for repeated named roles. A proposed
   token must name two independent consumers or a declared theme/context
   substitution; keep a one-off color, spacing, radius, or size local. Keep
   primitive values beneath functional roles and use component/pattern tokens
   only for genuine repeated specialization. Never let a raw palette, spacing
   scale, or component kit determine content priority.
8. Implement the representative fragment through the real stack, render it,
   and inspect its strongest intended reading and most plausible wrong reading.
   Revise it before propagating the relation to the rest of the coherent
   vertical slice. Preserve semantic structure, keyboard reachability,
   readable contrast, useful focus,
   reduced-motion behavior, source boundaries, and existing functional
   contracts. These are production conditions, not proof of beauty.
9. Search the whole project—not only the nearest source directory—for
   consumers of every changed shared role, token, asset, metadata value, or
   component contract. Inspect representative routes, states, content lengths,
   themes, and viewports, including the highest-risk real content condition
   such as a writing system, dense record, empty state, or user-generated
   extreme when it changes the design judgment. Check both the strongest
   designed surface and the ordinary downstream surface most likely to reveal
   a broken system or attention path. For motion, inspect initial, transition,
   settled, and reduced-motion states rather than only the final frame.
10. Remove temporary probes or dependencies that do not belong to the product,
    then run the relevant mechanical checks. Return a compact implementation
    account:

```text
Object, audience, action, and inherited direction:
Decisive relation, negative boundary, and representative fragment:
What the first render confirmed, rejected, or changed:
Implemented structure, component expression, and deliberately neutral details:
Shared roles and project-wide consumers checked:
Representative routes, states, content, themes, and viewports inspected:
Mechanical verification and known limits:
Remaining aesthetic judgment and the observation that would require redesign:
```

Do not promote the implementation's palette, typography, grid, components, or
motion to a reusable project direction from this run alone.
