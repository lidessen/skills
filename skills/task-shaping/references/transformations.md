# Task-shaping transformations

Load this reference only after naming a concrete profile/task mismatch. Choose
one smallest useful move; do not apply the rows as a standard workflow.

| Observed pressure | Candidate transformation | Whole that must remain visible | Failure or reopening signal |
|---|---|---|---|
| One coherent task already settles correctly in comparable runs | direct execution | original acceptance and authority | added packets create coordination without reducing error or overload |
| Evidence navigation consumes reasoning context | typed retrieval plus runtime-bounded expansion | source revision, provenance, exact-value and authority reopen path | evidence arrives but the judgment still skips or misuses it |
| Agent must notice every item in a finite claim or requirement set | externalized ledger or addressable audit queue with required per-item disposition | completeness set and acceptance rule | generic “checked all sections” output still permits omissions |
| Free scalar score has no stable calibration | repeated pairwise choice under one operational criterion, with tie/abstain and mechanical aggregation | separate criteria, presentation-order balance, and human decision boundary | pairwise bias is correlated, aggregation hides disagreement, or score language returns |
| Several locally testable semantic regions overload one context | partition by coherent question, causal path, state/effect owner, contract, or evidence domain | low-resolution whole, incoming/outgoing relations, deliberate overlap | arbitrary file cuts sever meaning or synthesis rediscovers boundaries |
| One loop must explore many independent branches | external branch queue with retained settlements | branch-opening condition and parent obligation | successful branches rerun or failures disappear from aggregate state |
| Work spans sessions or external events | externalize state, checkpoints, source revisions, and resume conditions | current decision state and invalidation signals | the next loop reconstructs state from prose or stale summaries |
| Semantic judgment and irreversible effect occur in one loop | separate prepare, verify, authorize, and commit | intended effect, rollback, authority, and evidence | executor self-accepts or verification occurs after an unrecoverable effect |
| A generated result is locally valid but global relations remain uncertain | independent reconstruction packet with coverage and boundary ledger | all obligations, conflicts, missing relations, and source status | reducer concatenates, votes, or rereads every raw input |
| Reconstructor or verifier itself exceeds the envelope | shape its obligations into claim-addressed or relation-addressed units | candidate identity and final acceptance question | meta-level split grows without a bounded settlement path |
| No comparable capability evidence exists | smallest discriminatory discovery probe, then `model-evaluation` for reusable claims | proposed allocation and disconfirming outcome | task is partitioned from prestige, token count, or guesswork |
| Local unit remains unstable after a truthful minimum shape | escalate model/profile or route to human judgment | failed evidence and unchanged hard constraints | repeated retries consume work without changing task form |

## Semantic partition test

A proposed unit should answer yes to these questions:

- Does it contribute one necessary determination to the original outcome?
- Can its local result be accepted or rejected from bounded evidence?
- Are relations crossing its boundary named rather than silently assumed?
- Can it report overload or `unverifiable` without expanding authority?
- Can a later actor reconstruct its contribution without reopening its entire
  raw work record?

If not, merge it with the relation it needs, reshape its question, or leave the
task direct.

## Adaptive resolution

Treat unit granularity as an observed control variable, not a fixed token or
file-size rule:

- reduce the unit when it cannot state one judgment, repeatedly misses local
  obligations, exhausts its loop on navigation, or shows high result variance;
- enlarge or merge units when boundary traffic, duplicated context, conflict,
  or reconstruction work costs more than the local reasoning avoided; and
- keep coupled work together when order, shared mutable state, or a non-local
  invariant prevents independent settlement.

Record the observation that changed granularity. “More Cells” is not evidence
of a better approximation.

## Reconstruction test

Before releasing units, simulate the reducer's work:

1. map every global obligation to units;
2. identify duplicated boundaries and the source that can resolve conflict;
3. name missing-relation and stale-revision behavior;
4. bound the evidence the reducer needs by default;
5. state when one targeted follow-up is allowed; and
6. verify that acceptance remains outside the producing units.

Also test integrability: source revisions agree, order-dependent effects remain
ordered, duplicated boundary values reconcile, cross-unit interactions are
represented, and accumulated local uncertainty stays visible.

If these steps require the reducer to reproduce every local investigation, the
partition is not yet a smaller executable form.
