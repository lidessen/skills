# Task-shaping evaluation

Evaluate whether the transformation changes agent behavior, not whether the
Task Shape sounds systematic.

## Minimum probes

| Probe | Question | Disconfirming observation |
|---|---|---|
| Action | Does a task outside the observed envelope become locally settleable and reconstructible? | units still omit material obligations, exceed the envelope, or move the whole problem into synthesis |
| Direct boundary | Does the Skill keep one coherent, low-risk task direct? | it manufactures packets, ledgers, or orchestration for a task already inside the envelope |
| Ownership boundary | Does it prepare semantic work without choosing provider, concurrency, budget, execution, or acceptance? | the Task Shape becomes a runtime manifest, resource decision, or self-approved result |
| Context | Does each unit receive the minimum evidence that changes its judgment while global obligations remain reachable? | irrelevant whole-project context loads everywhere, or a hard constraint disappears from local work |
| Locality | Does a bounded unit distinguish packet absence from whole-source absence? | it reports a missing file, correction, caller, or invariant merely because that evidence is outside its packet |
| Evidence cost | Is cumulative delivered context measured across loop steps, including repeated and cached input? | an initial packet-size estimate hides repeated rereads or re-injection that dominates total work |
| Promotion | Does a proposed guard remain a hypothesis until matched evidence shows it contains the known error within an accepted tolerance? | the Skill labels a guarded operation reliable from instructions or a new schema alone |
| Evidence status | Does absent comparable evidence remain `discovery-needed` while the execution disposition uses the declared four-state contract? | the result invents a fifth disposition or guesses a primitive from model reputation |
| Domain ownership | When another method owns semantic partitioning, does Task Shaping stop at envelope and handoff constraints? | it emits example semantic units and then disclaims ownership in a routing note |

For the promotion probe, include at least one tempting near-success whose
observed errors violate a zero-tolerance obligation. The correct disposition is
`guarded` even when the requested transformation proposes an apparently apt
schema, abstention rule, or verifier. Pair it with a truly matched retained case
that met tolerance so the probe tests both refusal and justified promotion.

## Comparative claim

When claiming improved stability, keep the original outcome, source revision,
execution profile, evidence access, acceptance, and evaluator fixed. Compare the
unchanged task with the shaped form over repeated runs. Observe at least:

- protocol settlement separately from material correctness;
- omitted obligations and unsupported claims;
- cross-unit conflicts and repair;
- synthesis or verification work;
- retries, token use, latency, and human repair;
- cumulative input delivery, including duplicated packet material and cache
  behavior across model steps; and
- failures that remain detectable rather than silently passing.

Development cases that teach a transformation do not confirm it. Retain one or
more different cases from the claimed task family before promoting a reusable
shape or capability boundary.

## Result record

```text
Original task and execution profile:
Observed envelope mismatch:
Unchanged form and result:
Transformation and retained global obligations:
Unit and reconstruction results:
Material correctness and completeness:
Work, variance, and repair observations:
Verdict: supported | failed | inconclusive
Boundary learned and next reopening observation:
```

A structural schema check proves only that a Task Shape can be represented. A
successful terminal call proves only protocol settlement. Neither proves that
the transformation preserved meaning or improved the original decision.
